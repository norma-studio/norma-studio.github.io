В этом уроке мы рассмотрим WebDriverJs, который является инструментом, используемым для автоматизации браузера. Chrome будет использоваться во всех современных браузерах, как правило, есть доступные драйверы для использования с WebDriver (даже мобильными), поэтому проверьте их, если вы хотите автоматизировать другие браузеры.

В то время как модульные тесты, безусловно, ценны для современных веб-приложений, в какой-то момент, когда ваше приложение растет, вы обнаружите ошибки, которые не были пойманы модульным тестом, но теоретически были бы пойманы тестом интеграции/приемочным тестированием.

Если вы хотите следовать стратегии тестирования, которая включает в себя проверку в браузере, это руководство даст вам начальное введение в тестирование с помощью WebDriverJs, чтобы вы получили достаточное количество знаний для начала работы.

В этом руководстве предполагается, что вы знакомы с JavaScript и можете запускать код JavaScript с помощью node.js.

WebDriverJS

Если вы хотите следовать статье, не стесняйтесь проверить этот образец проекта, который содержит несколько примеров WebDriver для запуска. Вам также потребуется установить Chromedriver и сделать его доступным в вашем path.

У Selenium WebDriver обычно есть сервер и клиент. Помимо разработчиков WebDriver, большинство людей будут интересоваться только клиентским API, который позволяет им управлять браузером через их сценарий. Чтобы начать работу, установите привязки JavaScript для WebDriver:

npm install selenium-webdriver

После того, как вы установили этот модуль через NPM, вам может потребоваться модуль в вашем коде node следующим образом:

require("selenium-webdriver");

Кроме того, если вы посмотрите пример проекта, вы можете просто запустить npm install внутри папки, поскольку модуль WebDriverJs указан как зависимость в файле package.json.

Пока вы можете просматривать официальную документацию, мне больше всего нравится смотреть исходный код. В файле webdriver.js перечислены многие методы WebDriver, например. вы заметите getAttribute и getText. Вот некоторые методы, которые могут представлять интерес:

  • get - перемещение браузера по URL-адресу.
  • findElements - похоже на document.querySelectorAll в браузере.
  • executeScript - выполнить необработанный JavaScript на текущей странице.
  • getText - Получить текстовое содержимое элемента, включая его дочерние элементы.
  • isDisplayed - узнать, отображается ли элемент на странице.

Обещания

Одним из факторов, связанных с привязками JavaScript для WebDriver, является то, что почти каждый метод является асинхронным. Это означает, что следующий код фактически не получает заголовок веб-страницы:

var title = browser.getTitle();//logs { then: [Function: then],  cancel: [Function: cancel], isPending: [Function: isPending] }console.log(title);

Вместо этого вам нужно сделать следующее:

var promise = browser.getTitle();promise.then(function(title) {    console.log(title);});

Это связано с тем, что WebDriverJs использует обещания, чтобы немного улучшить внешний вид асинхронного кода. Обратите внимание, что реализация обещаний как часть WebDriverJs не соответствует точно стандарту Promises/A +.

Главное, что нужно здесь понять, это то, что большинство методов WebDriver возвращают метод then, который принимает два необязательных (функциональных) аргумента. Первый аргумент - это обратный вызов, который может принимать значение.

В приведенном выше примере мы попросили заголовок, поэтому наш обратный вызов получит этот заголовок в качестве его первого аргумента. Второй необязательный аргумент функции, который мы можем передать методу then, позволяет нам ловить ошибки, если вообще они были.

Примеры

Давайте вспомним, где мы находимся:

  1. Мы установили двоичный файл Chromedriver.
  2. Мы установили WebDriverJs через NPM.
  3. С пониманием, что почти все происходит асинхронно, мы знаем, как использовать обещания для получения желаемых значений.

Посмотрите на этот пример кода:

var webdriver = require("selenium-webdriver");var browser = new webdriver.Builder().usingServer().withCapabilities({"browserName": "chrome" }).build();browser.get("http://en.wikipedia.org/wiki/Wiki");browser.findElements(webdriver.By.css("[href^="/wiki/"]")).then(function(links){    console.log("Found", links.length, "Wiki links." )    browser.quit();});

Запустите пример Wiki следующим образом:

$ node Wiki.jsFound 367 Wiki links.

В примере кода первые несколько строк являются, по существу, шаблонами. Он инициализирует объект браузера и указывает некоторую начальную конфигурацию, например, какой браузер фактически используется. Начиная с вызова в browser.get, у нас есть код, который наc действительно волнует.

  1. Сначала переходим на страницу Википедии.
  2. Мы строим селектор CSS, который соответствует элементам, которые имеют атрибут href и значение, начинающееся с /wiki/ (например, внутренние ссылки Wiki).
  3. Все еще в той же строке, что и в шаге №2, мы передаем селектор CSS в метод findElements, который будет асинхронно оценивать выражение селектора.
  4. Чтобы наблюдать обновления Promise, мы передаем функцию обратного вызова методу then.
  5. Первый аргумент обратного вызова - это массив согласованных элементов, поэтому мы извлекаем это и выводим его длину.
  6. Наконец, мы выходим из браузера.

Поиск элементов на странице - это одна часть головоломки, давайте взглянем на другой пример, демонстрирующий выполнение поиска Google и нажатие на результат, который мы ожидаем на странице.

/** Carry out a Google Search*/"use strict";var webdriver = require("selenium-webdriver");var browser = new webdriver.Builder().usingServer().withCapabilities({"browserName": "chrome" }).build();function logTitle() {    browser.getTitle().then(function(title) {        console.log("Current Page Title: " + title);    });}function clickLink(link) {    link.click();}function handleFailure(err) {    console.error("Something went wrong\n", err.stack, "\n");    closeBrowser();}function findTutsPlusLink() {    return browser.findElements(webdriver.By.css("[href="http://code.tutsplus.com/"]")).then(function(result) {        return result[0];    });}function closeBrowser() {    browser.quit();}browser.get("https://www.google.com");browser.findElement(webdriver.By.name("q")).sendKeys("tuts+ code");browser.findElement(webdriver.By.name("btnG")).click();browser.wait(findTutsPlusLink, 2000).then(clickLink).then(logTitle).then(closeBrowser, handleFailure);

Выполнение вышеуказанного кода:

$ node GoogleSearch.jsCurrent Page Title: Tuts+ Code Tutorials

Здесь показаны несколько интересных фрагментов. Во-первых, мы можем понять, каково это - использовать объявления функций - вместо анонимных обратных вызовов функции (которые передаются в then), результат - это что-то вроде текучего API (см. последнюю строку). Кроме того, поскольку у нас есть возможность создавать пользовательские обещания (отложенные), мы можем иметь такой же текучий интерфейс!

Обратите внимание, мы подключаем колбэк для ошибки к последнего вызову then, даже если ошибка происходит раньше, она все равно будет распространяться по цепочке.

Перейдите на главную страницу Google и найдите «tuts + code». Поскольку мы работаем над объектом браузера, внутренний механизм управления потоком WebDriver знает, что каждая команда должна выполняться одна за другой, это избавляет нас от необходимости объединять все вместе, а также объясняет, почему существует два вызова findElement, один после другого, без необходимости сцеплять их друг к другом.

Ожидание

Когда мы выполняем поиск Google с домашней страницы, перезагрузка страницы не происходит, поэтому WebDriver немедленно попытается найти элементы, которые мы проинструктировали на странице результатов поиска. Знание то, когда ждать элементы, является важной частью автоматизации браузера.

Старый и наивный способ это сделать - это использовать функцию sleep. Поскольку точка, в которой появляется элемент, может сильно зависеть от внешних факторов (например, скорости сетевого подключения), разработчики могут иногда указывать WebDriver ждать определенного периода времени, прежде чем продолжить. Это, конечно, пронизано проблемами.

К счастью, метод wait упрощает автоматизацию современных веб-страниц. Вы вызываете wait с двумя аргументами, первая - это функция, которая должна оценивать значение true на период времени, определяемый как второй аргумент для ожидания. WebDriver регулярно вызывает ваш обратный вызов до тех пор, пока он не вернет true, или время не закончилось, и в этом случае возникает ошибка.

Изменение браузера

Хотя существует множество способов вызова контекста элементов DOM, вы также можете вызвать методы в самом браузере, чтобы дать вам больше контроля над состоянием браузера. Вот несколько простых примеров, которые помогут вам лучше это понять:

Задать размеры окна браузера

browser.manage().window().setSize(1280, 720)

Подключить браузер к прокси:

var proxy = require("selenium-webdriver/proxy");browser = new webdriver.Builder().usingServer().withCapabilities({"browserName": "chrome" }).setProxy(proxy.manual({    http: "127.0.0.1:9000"})).build();

Вы также можете читать, писать и удалять файлы cookie, делать скринщот, устанавливать отдельные настройки браузера и многое другое.

Альтернативные опции

Существует множество опций, когда вы хотите программно управлять браузером. Прежде всего, мы рассмотрели привязки JavaScript для WebDriver, но есть и другие:

  • Java
  • Python
  • Ruby
  • PHP

WebDriverJs, например. версия, которую мы установили с помощью npm install selenium-webdriver, - это всего лишь одна версия API клиента WebDriver, написанная на JavaScript. Если вы хотите программно управлять браузерами через JavaScript, есть и другие варианты:

  • WD.js - Fluent API с использованием обещаний + чепочки.
  • Leadfoot - теперь используется последней версией Intern.
  • WebDriver.io - Имеет кучу документации для использования с базами данных BDD/TDD.
  • Testium - имеет четкую документацию о том, что именно поддерживается.
  • DalekJS - веселый веб-сайт с хорошими отзывами при выполнении тестов. Многие DalekJS были разделены на модули, которые можно найти на Github.
  • Nightwatch - еще один инструмент с красивой обратной связью и плавным API.
  • Webdriver-sync - Синхронная версия взаимодействия с WebDriver.

Использование чего-то типа WD.js или Nightwatch может означать ряд вещей:

  • Различные API для взаимодействия. Если у официальных ссылок JavaScript selenium-webdriver есть API, к которым вы не привыкли, ознакомьтесь с альтернативными вариантами выше.
  • Альтернативная обратная связь - это может быть на уровне репортера, но также и то, что вы видите в терминале после того, как тест не удался локально.
