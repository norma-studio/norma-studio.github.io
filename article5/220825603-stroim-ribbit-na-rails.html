Добро пожаловать в следующий выпуск в нашей серии клонов Twitter! В этом уроке мы построим Ribbit с нуля, не используя PHP, а с Ruby on Rails. Давайте начнем!

Одно небольшое объявление , прежде чем мы начнем: мы не будем разрабатывать пользовательский интерфейс для приложения в этом учебнике; это было сделано в Build the Twitter Clone From Scratch: The Design. Я дам вам знать, нужно ли нам что-то менять из этой статьи.

Шаг 0: настройка среды

Прежде всего: я использую Ruby 1.9.3 (p194) и Rails версии 3.2.8 для этого урока. Убедитесь, что вы используете те же версии. Неважно, как вы устанавливаете Ruby; вы можете использовать RVM (учебник), rbenv или просто обычную установку Ruby. Независимо от подхода, каждый установщик предоставляет вам двоичный файл gem, который затем можно использовать для установки Rails. Просто используйте эту команду:

    gem install rails

Это устанавливает последнюю версию Rails, и мы можем начать создавать наше приложение сейчас, когда все установлено. Надеюсь, вы понимаете, что Rails - это инструмент командной строки; вам нужно будет чувствовать себя комфортно в терминале, чтобы работать комфортно с этим учебником.

Шаг 1. Создание приложения Rails

Начнем с создания проекта. В командной строке перейдите в любой каталог, где вы хотите чтобы находился новый проект. Затем запустите:

    rails new ribbitApp

Эта одиночная команда создает несколько файлов внутри папки, называемой ribbitApp. Это то, что дает нам Rails; он даже установил gem"ы, необходимые для проекта.

Давайте cd в этот каталог и инициализируем git repo.

    cd ribbitApp    git init

Один из файлов, созданных в рельсах, - .gitignore. Если вы находитесь на Mac, вы, вероятно, захотите добавить в этот файл следующую строку - просто чтобы все было чисто:

    **.DS_Store

Теперь мы готовы совершить первый коммит!

    git add .    git commit -m "initial rails app"

Шаг 2. Подготовка пользовательского интерфейса.

Учебник интерфейса познакомил вас с изображениями и таблицами стилей Ribbit. Загрузите эти ресурсы и скопируйте файлы и папки gfx и less.js и style.less в каталог public вашего приложения.

Давайте добавим правило в style.less: стиль для наших флеш-сообщений. Вставьте это в нижней части файла:

    .flash {        padding: 10px;        margin: 20px 0;        &.error {            background: #ffefef;            color: #4c1717;            border: 1px solid #4c1717;        }        &.warning {            background: #ffe4c1;            color: #79420d;            border: 1px solid #79420d;        }        &.notice {            background: #efffd7;            color: #8ba015;            border: 1px solid #8ba015;        }    }

Это все! Давайте сделаем еще один коммит:

    git add .    git commit -m "Add flash styling"

Теперь давайте создадим макет. Это HTML, который обертывает основное содержимое каждой страницы - по существу, это верхний и нижний колонтитулы. Приложение Rails хранит это в app/views/layouts/application.html.erb. Избавьтесь от всего в этом файле и добавьте следующий код:

    <!DOCTYPE html>    <html>    <head>        <link rel="stylesheet/less" href="/style.less">        <script src="/less.js"></script>    </head>    <body>        <header>            <div class="wrapper">                <img src="/gfx/logo.png">                <span>Twitter Clone</span>            </div>        </header>        <div id="content">            <div class="wrapper">                <% flash.each do |name, msg| %>                    <%= content_tag :div, msg, class: "flash #{name}" %>                <% end %>                <%= yield %>            </div>        </div>        <footer>            <div class="wrapper">                Ribbit - A Twitter Clone Tutorial<img src="/gfx/logo-nettuts.png">            </div>        </footer>    </body>    </html>

Есть три вещи, которые вы должны заметить об этом. Во-первых, каждый URL-адрес публичного ресурса (изображения, таблицы стилей, JavaScript) начинается с косой черты (/). Это значит, что мы все еще можем загружать ресурсы, когда мы находимся на «более глубоких» маршрутах. Во-вторых, обратите внимание на разметку для отображения флэш-сообщений. Это отображает флэш-сообщения, когда они существуют. И в-третьих, обратите внимание на <% = yield%>; здесь мы вставляем другие «под»-шаблоны.

Хорошо, давайте сделаем следующее:

    git add .    git commit -m "Edit application.html.erb"

Шаг 3: Создание пользователей

Конечно, мы не можем иметь клон Twitter без пользователей, поэтому давайте добавим эту функциональность. Это может быть сложной функцией, но c Rails сделать это гораздо проще.

Естественно, мы не хотим хранить пароли наших пользователей как обычный текст; это огромный риск для безопасности. Вместо этого мы будем полагаться на Rails для автоматизации процесса шифрования наших паролей. Начнем с открытия нашего Gemfile и поиска этих строк:

    # To use ActiveModel has_secure_password    # gem "bcrypt-ruby", "~> 3.0.0"

Использование has_secure_password - это именно то, что мы хотим сделать, поэтому рас-комментируем эту вторую строку. Сохраните файл и вернитесь в командную строку. Теперь нам нужно запустить bundle install для установки gem bcrypt.

Мы можем создать наш пользовательский ресурс, как только gem будет установлен.

Ресурс Rails - это в основном модель, связанный с ней контроллер и несколько других файлов.

Мы создаем ресурс с помощью команды rails generate (или rails g):

    rails generate resource user username name email password_digest avatar_url

Мы передаем несколько параметров этой команде, в результате получим resource, называемый user; его модель имеет следующие пять полей:

  • username: уникальное имя пользователя, эквивалентное дескриптору Twitter.
  • name: фактическое имя пользователя.
  • email: их адрес электронной почты.
  • password_digest: зашифрованная версия их пароля.
  • avatar_url: путь к изображению аватара.

Теперь нам нужно мигрировать нашу базу данных, чтобы она была настроена для хранения пользователей. Мы выполнили это с помощью следующей команды:

    rake db:migrate

Это создает таблицу users, но мы напрямую не взаимодействуем с базой данных с Rails. Вместо этого мы используем ORM ActiveRecord. Нам нужно добавить код в app/models/user.rb. Откройте этот файл.

Когда Rails сгенерировал этот файл, он добавил вызов метода attr_accessible. Этот метод определяет, какие свойства читаются и записываются в экземпляры этого класса. По умолчанию все вышеупомянутые свойства доступны, но мы хотим изменить это:

    attr_accessible :avatar_url, :email, :name, :password, :password_confirmation, :username

Большинство из них должно иметь смысл, но что такое password и password_confirmation? В конце концов, у нас есть только поле password_digest в нашей базе данных. Это часть магии Rails, о которой я упоминал ранее. Мы можем установить поля password и password_confirmation на экземпляре User. Если они совпадают, пароль будет зашифрован и сохранен в базе данных. Но чтобы подключить эту функциональность, нам нужно добавить еще одну строку в наш класс модели User:

    has_secure_password

Затем мы хотим включить валидацию в нашу модель. Вызов has_secure_password обрабатывает поля пароля, поэтому мы будем обрабатывать поля email, username и name.

Поле name простое: мы просто хотим убедиться, что оно присутствует.

    validates :name, presence: true

Для поля username мы хотим убедиться, что оно существует и является уникальным; ни один из двух пользователей не может иметь одинаковое имя пользователя:

    validates :username, uniqueness: true, presence: true

Наконец, поле email не только должно существовать и быть уникальным, но также должно соответствовать регулярному выражению:

    validates :email, uniqueness: true, presence: true, format: { with: /^[\w.+-]+@([\w]+.)+\w+$/ }

Это очень упрощенное регулярное выражение электронной почты, но оно должно работать для наших целей.

Теперь, как насчет этого поля avatar_url? Мы хотим использовать адрес электронной почты пользователя и вытащить связанный с ним Gravatar, поэтому нам нужно создать этот URL. Мы могли бы сделать это «на лету», но сохранить его в базе данных будет более эффективно. Во-первых, нам нужно убедиться, что у нас есть чистый адрес электронной почты. Давайте добавим метод к нашему классу User:

privatedef prep_email    self.email = self.email.strip.downcase if self.emailend

Это ключевое слово private означает, что все методы, определенные после ключевого слова, определяются как частные методы; они не могут быть доступны извне класса (по экземплярам).

Этот метод prep_email обрезает пробелы в начале и конце строки, а затем преобразует все символы в нижний регистр.

Это необходимо, потому что мы собираемся генерировать хэш для этого значения.

Мы хотим, чтобы этот метод выполнялся непосредственно перед процессом валидации; добавьте следующую строку кода в верхней части класса. 

    before_validation :prep_email

Затем давайте создадим URL-адрес аватара, написав другой метод (поставьте его под одним из выше): 

    def create_avatar_url        self.avatar_url = "http://www.gravatar.com/avatar/#{Digest::MD5.hexdigest(self.email)}?s=50"    end

Нам нужно вызвать этот метод, прежде чем мы сохраним пользователя в базе данных. Добавьте этот вызов в начало файла:

    before_save :create_avatar_url

Вот и все! Теперь у нас есть механика нашей пользовательской части. Прежде чем писать пользовательский интерфейс для создания пользователей, давайте закомитим нашу работу:

    git add .    git commit -m "Create user resource"

Шаг 5: Написание пользовательского интерфейса

Создание пользовательского интерфейса в Rails означает, что нам нужно знать маршруты, которые отображают наши представления. Если вы откроете config/routes.rb, вы найдете такую строку:

    resources :users

Это было добавлено в файл routes.rb, когда мы сгенерировали ресурс пользователя, и он устанавливает маршруты REST по умолчанию. Прямо сейчас, маршрут, который нам интересен, это маршрут, который отображает форму для создания новых пользователей: /users/new. Когда кто-то отправится на этот маршрут, будет выполнен метод new в контроллере user, так что начнем.

Пользовательский контроллер находится в app/controllers/users_controller.rb. По умолчанию у него нет методов, поэтому давайте добавим метод new в класс UsersController.

    def new        @user = User.new    end

Как вы знаете, переменные экземпляра Ruby начинаются с @ - делая @user доступным изнутри нашего представления. Перейдем к представлению, создав файл с именем new.html.erb в папке app/views/users. Вот что происходит в этом отображении:

<img src="/gfx/frog.jpg"><div class="panel right">  <h1>New to Ribbit?</h1>    <%= form_for @user do |f| %>    <% if @user.errors.any? %>      <ul>        <% @user.errors.full_messages.each do |message| %>          <li><%= message %></li>        <% end %>      </ul>    <% end %>    <%= f.text_field :email, placeholder: "email" %>    <%= f.text_field :username, placeholder: "username" %>    <%= f.text_field :name, placeholder: "name" %>    <%= f.password_field :password, placeholder: "password" %>    <%= f.password_field :password_confirmation, placeholder: "password" %>    <%= f.submit "Create Account" %>  <% end %></div>

Это фактически представление, которое служит представлением домашней страницы, когда пользователь не вошел в систему. Мы используем метод form_for для создания формы и передаем ей переменную @user в качестве параметра. Затем внутри формы (которая находится внутри блока Ruby) мы сначала печатаем ошибки. Конечно, при первом посещении не будет никаких ошибок на странице.

Однако любой вход, который не соответствует нашим правилам валидации, приводит к сообщению об ошибке, которое отображается в элементе списка.

Затем у нас есть поля для наших пользовательских свойств. Поскольку в этом дизайне нет лейблов для текстовых полей, я поместил текст лейбла в качестве заполнителя полей (используя атрибут placeholder). Они не будут отображаться в старых браузерах, но это не относится к нашей главной цели.

Теперь, что происходит, когда пользователь нажимает кнопку «Создать учетную запись»? Эта форма будет POST на маршрут /users, что приведет к выполнению метода create в контроллере users. Вернемся к этому контроллеру, затем:

def create  @user = User.new(params[:user])  if @user.save    redirect_to @user, notice: "Thank you for signing up for Ribbit!"  else    render "new"  endend

Начнем с создания нового пользователя, который попадает в метод new в виде значений из нашей формы. Затем мы вызываем метод save. Этот метод сначала проверяет ввод; если данные находятся в правильном формате, метод вставляет запись в базу данных и возвращает значение true. В противном случае возвращается false.

Если @user.save возвращает true, мы перенаправляем зрителя на ... сам объект @user?

Это фактически перенаправляет на путь для этого пользователя, который будет /users/. Если @user.save возвращает false, мы повторно открывает путь /users/new и отображаем ошибки валидации. Мы также предварительно заполняем поля формы с ранее предоставленной информацией пользователя. Умно, да?

Итак, если мы направляем пользователя к их новому профилю пользователя, нам нужно создать следующую страницу. Это вызывает метод show в пользовательском контроллере, поэтому мы сперва добавим его:

def show  @user = User.find(params[:id])end

Этот метод просматривает номер идентификатора на маршруте (например, /users/4) и находит связанного пользователя в базе данных. Как и прежде, мы теперь можем использовать эту переменную @user из представления.

Создайте файл app/views/users/show.html.erb и добавьте следующий код:

    <div id="createRibbit" class="panel right">        <h1>Create a Ribbit</h1>        <p>        <form>            <textarea name="text" class="ribbitText"></textarea>            <input type="submit" value="Ribbit!">        </form>        </p>    </div>    <div id="ribbits" class="panel left">        <h1>Your Ribbit Profile</h1>        <div class="ribbitWrapper">            <img class="avatar" src="<%= @user.avatar_url %>">            <span class="name"><%= @user.name %></span> @<%= @user.username %>            <p>            XX Ribbits            <span class="spacing">XX Followers</span>            <span class="spacing">XX Following</span>            </p>        </div>    </div>    <div class="panel left">        <h1>Your Ribbits</h1>        <div class="ribbitWrapper">            Ribbits coming . . .        </div>    </div>

Вы заметите, что в этом представлении есть несколько заполнителей. Во-первых, есть форма для создания нового ribbit. Затем есть последователь, следующие числа и список ваших ribits. Мы скоро придем к этому.

На этом шаге еще одна вещь: мы хотим, чтобы корневой маршрут (/) показывал новую пользовательскую форму. Откройте файл config/routes.rb еще раз и добавьте эту строку:

Шаг 6: Добавление поддержки сеанса

Несмотря на то, что мы внедрили пользовательскую функцию, пользователь еще не может войти в систему. Итак, давайте добавим поддержку сессии.

Возможно, вы узнали, как мы создали учетные записи пользователей.

Я взял этот общий метод из эпизода Railscast 250. Этот эпизод также демонстрирует, как создать поддержку сеанса, и я буду использовать этот подход для Ribbit.

Начнем с создания контроллера для управления нашими сеансами. Мы фактически не будем хранить сеансы в базе данных, но нам нужно иметь возможность устанавливать и отменять переменные сеанса. Контроллер - это правильный способ сделать это.

    rails generate controller sessions new create destroy

Здесь мы создаем новый контроллер с именем sessions. Мы также создадим в нем методы new, create и destroy. Конечно, он не будет заполнять эти методы, но он создаст для нас свою «оболочку».

Теперь давайте откроем файл app/controllers/sessions_controller.rb. Метод new прекрасен, как есть, но метод create требует некоторого внимания. Этот метод выполняется после того, как пользователь вводит свои учетные данные и кликает «Войти». Добавьте следующий код:

    def create        user = User.find_by_username(params[:username])        if user && user.authenticate(params[:password])            session[:userid] = user.id            redirect_to rooturl, notice: "Logged in!"        else            flash[:error] = "Wrong Username or Password."            redirect_to root_url        end    end

Мы используем метод find_by_username в классе User для извлечения пользователя с предоставленным именем пользователя. Затем мы вызываем метод authenticate, передавая ему пароль. Этот метод был добавлен как часть функционала use_secure_password. Если учетные данные пользователя пройдут, мы можем установить переменную сеанса user_id в идентификатор пользователя. Наконец, мы перенаправляем на корневой маршрут с сообщением «Вы вошли!».

Если учетные данные пользователя не аутентифицируются, мы просто перенаправляемся на корневой маршрут и устанавливаем сообщение об ошибке «Имя пользователя или пароль ошибочны».

Выход из системы запускает метод destroy. Это очень простой метод:

    def destroy        session[:userid] = nil        redirect_to root_url, notice: "Logged out."    end

Этот код достаточно понятен; просто избавиться от этой переменной сеанса и перенаправить к корню.

Rails снова помогли нам и добавили три маршрута для этих методов. Вы можете найти их в config/routes.rb:

    get "sessions/new"    get "sessions/create"    get "sessions/destroy"

Мы хотим изменить маршрут sessions/create до POST, например:

    post "sessions/create"

Мы почти готовы добавить форму входа в наши представления. Но сначала давайте создадим вспомогательный метод, который позволит нам быстро получить зарегистрированного пользователя. Мы поместим это в контроллер приложения, чтобы мы могли получить к нему доступ из любого файла отображения. Путь к контроллеру приложения - app/controllers/application_controller.rb.

Шаг 9: Подписываемся на другого пользователя

Это не будет клон Twitter, если мы не сможем следовать за другими пользователями, поэтому давайте поработаем над этой функцией.

Сначала будет немного сложно. Подумайте об этом: наши записи User должны следовать другим записям User и за нами также могут следовать другие записи User. Это ассоциация «много-ко-многим», объединяющая себя. Это означает, что нам понадобится класс ассоциации, и мы будем называть его «Отношения». Начните с создания этого ресурса:

    rails g resource relationship follower_id:integer followed_id:integer    rake db:migrate

Эта модель требует только двух полей: идентификатора последователя и идентификатора followed (примечание: терминология здесь может немного запутывать. В нашем случае я использую термин «следовать», чтобы означать, что на пользователя подписались).

Затем мы хотим связать модель User с этой моделью Relationship, и мы делаем это с обеих сторон. Во-первых, в классе Relationship (app/models/relationship.rb) мы хотим добавить эти две строки:

    belongs_to :follower, classname: "User"    belongs_to :followed, classname: "User"

Первая строка связывает запись User с полем follower_id, а вторая строка связывает запись User с полем follow_id. Важно включить имя класса, потому что Rails не может вывести класс из имен свойств («follower» и «follow»). Однако он может вывести правильные поля базы данных (follower_id и follow_id) из этих имен.

Теперь, в классе User (app/model/user.rb), мы должны сначала подключить каждую модель пользователя к связанным с ней отношениям:

    has_many :follower_relationships, classname: "Relationship", foreign_key: "followed_id"    has_many :followed_relationships, classname: "Relationship", foreign_key: "follower_id"

Нам нужно создать две ассоциации, потому что у нас есть два набора отношений для каждого пользователя: всех людей, следующих за ними, и всех людей, которых на они подписаны. И нет, эти внешние ключи не должны переключаться. Ассоциация follower_relationship несет ответственность за всех ваших последователей. Следовательно, ему нужен внешний ключ follow_id.

Затем мы можем использовать эти отношения, чтобы добраться к фоловверам с другой стороны:

    has_many :followers, through: :follower_relationships    has_many :followeds, through: :followed_relationships

Это дает нашим пользователям методы followers и followeds. Эти оба метода возвращают массивы наших последователей или людей, за которыми мы следуем, соответственно.

Наконец, давайте добавим два метода в нашу пользовательскую модель, что поможет нам с пользовательским интерфейсом:

def following? user    self.followeds.include? userenddef follow user    Relationship.create follower_id: self.id, followed_id: user.idend

Давайте сделаем эти изменения перед настройкой пользовательского интерфейса.

    git add .    git commit -m "created user relationships infrastructure"

Теперь пользовательский интерфейс находится на страницах профиля пользователя, который является файлом app/views/users/show.html.erb. Начнем с чего-то простого: последователя и счетчика фолловеров. Смотрите, где у нас это?

    <span class="spacing">XX Followers</span>    <span class="spacing">XX Following</span>

Мы заменим эти значения-заполнители, например:

    <span class="spacing"><%= @user.followers.count %> Followers</span>    <span class="spacing"><%= @user.followeds.count %> Following</span>

У нас есть кнопка follow/unfollow под этими счетчиками, но есть несколько состояний, которые нам нужно учитывать. Во-первых, мы не хотим показывать какую-либо кнопку, если пользователь либо просматривает свой профиль, либо не входит в систему. Во-вторых, мы хотим отобразить кнопку «Отменить подписку», если пользователь уже следует за владельцем этого профиля.

Ресурс Rails - это в основном модель, связанный с ней контроллер и несколько других файлов.

Здесь формы более сложные. Во-первых, если текущий пользователь уже следует за просмотренным пользователем, мы будем использовать form_tag для создания формы, которая ведет в relationship_path. Конечно, мы не можем забыть установить method как delete, потому что мы удаляем отношения.

Если текущий пользователь не следит за просмотренным пользователем, мы создадим form_for для текущего отношения. Мы просто используем скрытое поле, чтобы определить, к какому пользователю следовать.

Если вы обратили внимание, вы узнаете, что чего-то не хватает: возможность манипулировать экземпляром отношений из этого представления. Нам нужен экземпляр Relationship. Как ни странно, найти правильное место в коде для этой страницы немного сложно. В конце концов, каждая страница в нашем Rails-приложении должна основываться на методе в одном из наших контроллеров. Мы можем использовать свойство массив followeds и отображать его, чтобы извлекать идентификаторы пользователя. Затем мы также вводим идентификаторы текущего пользователя и, наконец, извлекаем ribits для этих пользователей.

Сохраним шаблон в app/views/users/buddies.html.erb; он очень похож на наш public шаблон ribbits:

    <% if current_user %>    <div id="createRibbit" class="panel right">        <h1>Create a Ribbit</h1>        <p>        <%= form_for @ribbit do |f| %>            <%= f.textarea :content, class: "ribbitText" %>            <%= f.submit "Ribbit!" %>        <% end %>        </p>    </div>    <% end %>    <div class="panel left">        <h1>Buddies" Ribbits</h1>        <% @ribbits.each do |ribbit| %>            <div class="ribbitWrapper">                <a href="<%= user_path ribbit.user %>">                <img class="avatar" src="<%= ribbit.user.avatar_url %>">                <span class="name"><%= ribbit.user.name %></span>                 </a>                @<%= ribbit.user.username %>                 <span class="time"><%= time_ago_in_words(ribbit.created_at) %></span>                <p> <%= ribbit.content %> </p>            </div>        <% end %>    </div>

Нам нужно сделать маршрут для этого метода, чтобы его использовать. Откройте config/routes.rb и добавьте следующее:

    get "buddies", to: "users#buddies", as: "buddies"

Теперь мы можем открыть /buddies и посмотреть эту страницу!

Однако есть еще кое-что, что мы хотим сделать. Если входящий пользователь переходит к корневому URL-адресу, мы должны перенаправить его в /buddies. Помните, что этот маршрут в настоящее время:

    def new        @user = User.new    end

Давайте изменим это на следующее:

    def new        if current_user            redirect_to buddies_path        else            @user = User.new        end    end

Мы также должны добавить ссылку на страницу друзей в app/views/layouts/application.html.erb:

    <%= link_to "Buddies" Ribbits", buddies_path %>

И теперь мы сделаем эти изменения:

    git add .    git commit -m "added buddies page"

Шаг 11 Развертывание в Heroku

Последний шаг - развернуть приложение. Мы будем использовать Heroku.

Последний шаг - развернуть приложение. Мы будем использовать Heroku. Я собираюсь предположить, что у вас есть учетная запись Heroku, и что вы установили инструментальную панель Heroku (инструменты командной строки).

Перед тем, как мы начнем, мы сталкиваемся с проблемой! Мы использовали базу данных SQLite, потому что Rails по умолчанию использует SQLite. Тем не менее, Heroku не использует SQLite; он использует PostgreSQL в качестве базы данных. Мы должны внести изменения в наш Gemfile, что фактически нарушает нашу локальную копию приложения (если вы не установили и не настроили сервер PostgreSQL). Вот мой компромисс: я покажу вам, как это сделать здесь, и вы можете поиграть с моей развернутой версией. Но вы не внесете изменения в свой местный проект.

К счастью, переход Rails на PostgreSQL очень прост. В нашем Gemfile есть строка, которая выглядит так:

    gem "sqlite"

Измените эту строку следующим образом:

    gem "pg"

Теперь мы должны установить этот gem локально, чтобы обновить Gemfile.lock. Мы делаем это, выполнив:

    bundle install

И мы делаем комит:

    git add .    git commit -m "updated Gemfile with Postgres"

Теперь мы можем создать приложение Heroku. В нашем каталоге проекта запустите:

    heroku create    git push heroku master

И наконец:

    heroku open

Это откроет браузер с вашим развернутым приложением Heroku. Вы можете поиграть с моей развернутой копией.
