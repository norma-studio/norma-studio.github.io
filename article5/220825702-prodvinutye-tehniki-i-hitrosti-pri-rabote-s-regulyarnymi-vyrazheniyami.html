Интро

Они обладают обширным набором инструментов, некоторые из них остаются неисследованными и их не так часто используют. Добавление комментариев

Иногда регулярные выражения могут быть сложными и нечитабельными. Регулярное выражение, которое вы написали сегодня, завтра может оказаться непонятным для вас, не смотря на то что вы создали его. Также как и в случае с программированием, хорошей идей будет добавлять комментарии, тем самым вы сможете добавить пояснения и улучшить читабельность регулярного выражения.

Вот пример, как проверить формат номера Соединённых Штатов.

preg_match("/^(1[-\s.])?(\()?\d{3}(?(2)\))[-\s.]?\d{3}[-\s.]?\d{4}$/",$number)

Читабельность значительно улучшится, добавь мы комментарии и немного свободного пространства.

preg_match("/^  (1[-\s.])?# optional "1-", "1." or "1"( \( )?# optional opening parenthesis\d{3}# the area code(?(2) \) )# if there was opening parenthesis, close it[-\s.]?# followed by "-" or "." or space\d{3}# first 3 digits[-\s.]?# followed by "-" or "." or space\d{4}# last 4 digits$/x",$number);

Давайте разделим код на куски.

$numbers = array("123 555 6789","1-(123)-555-6789","(123-555-6789","(123).555.6789","123 55 6789");foreach ($numbers as $number) {echo "$number is ";if (preg_match("/^(1[-\s.])?# optional "1-", "1." or "1"( \( )?# optional opening parenthesis\d{3}# the area code(?(2) \) )# if there was opening parenthesis, close it[-\s.]?# followed by "-" or "." or space\d{3}# first 3 digits[-\s.]?# followed by "-" or "." or space\d{4}# last 4 digits$/x",$number)) {echo "valid\n";} else {echo "invalid\n";}}/* prints123 555 6789 is valid1-(123)-555-6789 is valid(123-555-6789 is invalid(123).555.6789 is valid123 55 6789 is invalid*/

Хитрость заключается в использовании "x" модификатора в конце регулярного выражения. Это позволяет нам игнорировать пробелы в шаблоне, до тех пор пока они не будут экранироваться с (\s). Тем самым мы с легкостью можем добавить комментарии. Комментарий начинается с "#" и заканчивается на новой строке.

Использование каллбеков

В PHP preg_replace_callback() может быть использован для добавления функционала каллбека во время использования замен регулярного выражения.

Иногда вам нужно несколько замен. Если вызвать preg_replace() или str_replace() для каждого шаблона, строка будет парситься снова и снова.

Давайте взглянем на пример с шаблоном электронной почты.

$template = "Hello [first_name] [last_name],Thank you for purchasing [product_name] from [store_name].The total cost of your purchase was [product_price] plus [ship_price] for shipping.You can expect your product to arrive in [ship_days_min] to [ship_days_max] business days.Sincerely,[store_manager_name]";// assume $data array has all the replacement data// such as $data["first_name"] $data["product_price"] etc...$template = str_replace("[first_name]",$data["first_name"],$template);$template = str_replace("[last_name]",$data["last_name"],$template);$template = str_replace("[store_name]",$data["store_name"],$template);$template = str_replace("[product_name]",$data["product_name"],$template);$template = str_replace("[product_price]",$data["product_price"],$template);$template = str_replace("[ship_price]",$data["ship_price"],$template);$template = str_replace("[ship_days_min]",$data["ship_days_min"],$template);$template = str_replace("[ship_days_max]",$data["ship_days_max"],$template);$template = str_replace("[store_manager_name]",$data["store_manager_name"],$template);// this could be done in a loop too,// but I wanted to emphasize how many replacements were made

Обратите внимание, что все замены имеют между собой что-то общее. Это всегда строка окружённая квадратными скобками. Мы можем найти их одним регулярным выражением и обработать замену каллбек функцией.

Более эффективный способ сделать это с помощью каллбеков:

// ...// this will call my_callback() every time it sees brackets$template = preg_replace_callback("/\[(.*)\]/","my_callback",$template);function my_callback($matches) {// $matches[1] now contains the string between the bracketsif (isset($data[$matches[1]])) {// return the replacement stringreturn $data[$matches[1]];} else {return $matches[0];}}

Теперь строка в $template парсится регулярным выражением одни раз.

Жадное против ленивого

Перед тем, как начать объяснять данную концепцию, я должен показать вам пример. Давайте предположим, что мы пытаемся найти тег ссылки в html документе:

$html = "Hello <a href="/world">World!</a>";if (preg_match_all("/<a.*>.*<\/a>/",$html,$matches)) {print_r($matches);}

Результат будет, как предполагалось:

/* output:Array(    [0] => Array        (            [0] => <a href="/world">World!</a>        ))*/

Давайте изменим текст и добавим второй тег ссылки:

$html = "<a href="/hello">Hello</a><a href="/world">World!</a>";if (preg_match_all("/<a.*>.*<\/a>/",$html,$matches)) {print_r($matches);}/* output:Array(    [0] => Array        (            [0] => <a href="/hello">Hello</a>            [1] => <a href="/world">World!</a>        ))*/

Опять же, кажется, что всё работает как надо. Не обманывайте себя. Единственная причина по которой это работает - теги ссылок находятся на отдельных строках и по умолчанию PCRE находит совпадения в шаблоне расположенном на одной строке (больше информации, касательно модификатора "m"). Если нам попадутся два тега ссылок на одной строке, работать это не будет:

$html = "<a href="/hello">Hello</a> <a href="/world">World!</a>";if (preg_match_all("/<a.*>.*<\/a>/",$html,$matches)) {print_r($matches);}/* output:Array(    [0] => Array        (            [0] => <a href="/hello">Hello</a> <a href="/world">World!</a>        ))*/

На этот раз шаблон соответствует первому открывающему тегу и последнему закрывающему тегу, и всему между ними, вместо создания двух отдельных соответствий. Это связано с тем, что по умолчанию мы имеем дело с жадным "greedy" режимом.

"Когда мы имеем дело с жадным режимом, квантификатор (такой как * или +) работает со всеми символами, который может найти."

Если вы добавите знак вопроса после квантификатора (.*?) он станет ленивым "ungreedy":

$html = "<a href="/hello">Hello</a> <a href="/world">World!</a>";// note the ?"s after the *"sif (preg_match_all("/<a.*?>.*?<\/a>/",$html,$matches)) {print_r($matches);}/* output:Array(    [0] => Array        (            [0] => <a href="/hello">Hello</a>            [1] => <a href="/world">World!</a>        ))*/

Теперь результат правильный. Другой способ переключиться в ленивый режим - использовать модификатор шаблона U.

Смотрящие вперёд и смотрящие назад утверждения

Смотрящие вперёд утверждения (lookhead assertion) ищут совпадения в шаблоне, которые соответствуют текущему совпадению. Объяснить это проще на примере.

Следующий шаблон сначала соответствует "foo", после чего проверят совпадает он или нет "bar":

$pattern = "/foo(?=bar)/";preg_match($pattern,"Hello foo"); // falsepreg_match($pattern,"Hello foobar"); // true

Это может показаться не очень полезным, так как вместо этого мы можем проверить "foobar". Однако, также у нас есть возможность воспользоваться смотрящим вперёд утверждением для поиска отрицательных утверждений. Следующий пример соответствует "foo", только в том случае, если это не "bar".

$pattern = "/foo(?!bar)/";preg_match($pattern,"Hello foo"); // truepreg_match($pattern,"Hello foobar"); // falsepreg_match($pattern,"Hello foobaz"); // true

Утверждения смотрящие назад работают похожим образом. но они ищут шаблон, до текущего соответствия. Вы можете применить (?< для положительных утверждений и (?<! для отрицательных утверждений.

Следующий шаблон совпадает в том случае если это "bar", но не "foo".

$pattern = "/(?<!foo)bar/";preg_match($pattern,"Hello bar"); // truepreg_match($pattern,"Hello foobar"); // falsepreg_match($pattern,"Hello bazbar"); // true

Условные (If-Then-Else) шаблоны

Регулярные выражения предоставляют функционал для проверки определённых условий. Формат выглядит следующим образом:

(?(condition)true-pattern|false-pattern)or(?(condition)true-pattern)

Условие может быть числом. В этом случае оно ссылается на предыдущий найденный подшаблон.

К примеру мы можем использовать это для проверки открывающих и закрывающих, угловых скобок:

$pattern = "/^(<)?[a-z]+(?(1)>)$/";preg_match($pattern, "<test>"); // truepreg_match($pattern, "<foo"); // falsepreg_match($pattern, "bar>"); // falsepreg_match($pattern, "hello"); // true

В примере выше, "1" ссылается на подшаблон (<), который также является опциональным, поскольку рядом стоит знак вопроса. Только в том случае если условие true, будет совпадение для закрывающей скобки.

Условие также может быть утверждением:

// if it begins with "q", it must begin with "qu"// else it must begin with "f"$pattern = "/^(?(?=q)qu|f)/";preg_match($pattern, "quake"); // truepreg_match($pattern, "qwerty"); // falsepreg_match($pattern, "foo"); // truepreg_match($pattern, "bar"); // false

Шаблоны фильтрации

Есть много разных причин фильтрации текста, во время разработки веб-приложений. Мы фильтруем данные перед тем как добавить их в базу данных, или перед тем как отобразить их в браузере. Таким же образом, необходимо фильтровать случайные строки, до того как включать их в регулярное выражение. PHP предоставляет функцию под называнием preg_quote для этого.

В следующем примере мы используем строку, которая содержит специальный символ (*).

$word = "*world*";$text = "Hello *world*!";preg_match("/".$word."/", $text); // causes a warningpreg_match("/".preg_quote($word)."/", $text); // true

Того же самого можно добиться закрыв строку между \Q и \E. Любой специальный символ после \Q будет игнорироваться до \E.

$word = "*world*";$text = "Hello *world*!";preg_match("/\Q".$word."\E/", $text); // true

Однако, второй метод не на 100% безопасен, так как строка может содержать \E.

Non-capturing подшаблоны

Подшаблоны, закрываемые скобками, будут добавлены в массив, тем самым мы можем использовать их позже, если понадобится. Также есть способ вовсе не сохранять их.

Давайте начнём с очень простого примера:

preg_match("/(f.*)(b.*)/", "Hello foobar", $matches);echo "f* => " . $matches[1]; // prints "f* => foo"echo "b* => " . $matches[2]; // prints "b* => bar"

Теперь давайте сделаем небольшое изменение, добавив подшаблон (H.*) для данного формата:

preg_match("/(H.*) (f.*)(b.*)/", "Hello foobar", $matches);echo "f* => " . $matches[1]; // prints "f* => Hello"echo "b* => " . $matches[2]; // prints "b* => foo"

Массив $matches был изменён, что в свою очередь повлечёт за собой неправильную работу скрипта, в зависимости от того, что мы делаем с переменными в коде. Теперь мы должны найти каждое вхождение массива $matches в коде и изменить соответствующим образом индекс.

Если нас не интересует контент нового подшаблона, который мы только что добавили, мы можем сделать его "non-capturing", вот так:

preg_match("/(?:H.*) (f.*)(b.*)/", "Hello foobar", $matches);echo "f* => " . $matches[1]; // prints "f* => foo"echo "b* => " . $matches[2]; // prints "b* => bar"

Добавив "?:" в начало подшаблона, мы не сохраняем совпадения в массив $matches, тем самым другие значения в массиве не будут сдвигаться.

Именованные подшаблоны

Есть другой способ для работы с подобными подводными камнями, как в предыдущем примере. Мы можем дать называние каждому подшаблону, далее мы сможем использовать их согласно их имени, вместо индексов в массиве. Формат для этого: (?Pшаблон)

Мы можем переписать первый пример из предыдущего раздела, вот так:

preg_match("/(?P<fstar>f.*)(?P<bstar>b.*)/", "Hello foobar", $matches);echo "f* => " . $matches["fstar"]; // prints "f* => foo"echo "b* => " . $matches["bstar"]; // prints "b* => bar"

Теперь мы можем добавить другой подшаблон, который не будет влиять на существующие совпадения в массиве $matches:

preg_match("/(?P<hi>H.*) (?P<fstar>f.*)(?P<bstar>b.*)/", "Hello foobar", $matches);echo "f* => " . $matches["fstar"]; // prints "f* => foo"echo "b* => " . $matches["bstar"]; // prints "b* => bar"echo "h* => " . $matches["hi"]; // prints "h* => Hello"

Не изобретайте колесо

Важно знать когда не стоит использовать регулярные выражения. Много ситуаций, при которых вы можете использовать другой функционал, другие утлиты.

Парсинг [X]HTML

Пост со Stackoverflow содержит отличное объяснение, почему мы не должны использовать регулярные выражения для парсинга [X]HTML.

...Господи помоги нам, как кто-либо может выжить во время такого бедствия, как использование регулярных выражений для парсинга HTML, в этом случае человечество обречено на бесконечные муки и дыры в безопасности, использование регулярных выражений в качестве инструмента для обработки HTML создаёт брешь между нашим миром и ужасной реальностью повреждённых сущностей...

Шутки в сторону, это хорошая идея потратить время и узнать какой XML или HTML парсер доступен и как он работает. К примеру, PHP предлагает много расширений относящихся к XML (и HTML).

Пример: Поиск второй ссылки на HTML странице

$doc = DOMDocument::loadHTML("<html><body>Test<a href="http://www.nettuts.com">First link</a><a href="http://net.tutsplus.com">Second link</a></body></html>");echo $doc->getElementsByTagName("a")->item(1)->getAttribute("href");// prints: http://net.tutsplus.com
