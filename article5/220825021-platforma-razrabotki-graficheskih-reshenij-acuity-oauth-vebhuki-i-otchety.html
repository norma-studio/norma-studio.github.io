OAuth2 интеграция Acuity Scheduling

Если вы расширяете приложение для работы с несколькими учетными записями Acuity, скажем, планируя большую компанию со многими учетными записями сотрудников Acuity, вы можете использовать интеграцию Acuity OAuth, чтобы каждый пользователь мог быстро и легко работать с календарями и управлять ими.

Вероятно, вы знакомы с OAuth при входе в приложение, используя свою учетную запись Facebook или Google. Acuity предоставляет эту же возможность для своих учетных записей (см. вышеприведенное изображение).

Как работает OAuth2

OAuth2 является стандартом для приложений, обеспечивающих надежную авторизацию и обмен учетными данными для API. Это лучший вариант, позволяющий пользователям вашего приложения подключаться к Acuity.

Пользователь нажимает кнопку подключения на клиенте (ваше приложение) и перенаправляется на Acuity для ввода своих учетных данных. После авторизации вашего приложения пользователь перенаправляется обратно в ваше приложение с помощью кода авторизации, который вы обменяете на токен доступа к API.

Чтобы начать работу, вам необходимо зарегистрировать учетную запись Acuity OAuth2:

URI обратного вызова является URL-адресом, где Acuity будет предоставлять код авторизации OAuth для запрашивающего пользователя. Затем вам понадобится использовать этот код и ваши ключи API для запроса постоянного токена для сохранения в вашей базе данных. Токен позволит вам выполнять расписания и вызовы API календаря от имени этого пользователя.

Вот пример URI обратного вызова:

https://your-app.example.com/oauth2-callback.php

Интеграция с Acuity OAuth2

После того, как вы получите учетную запись клиента, вы будете готовы начать интеграцию. Первым шагом является перенаправление пользователя на наш authorization_url. Обычно рекомендуется отображать кнопку типа «Подключиться к Acuity».

Например, вы можете связать пользователя с кнопкой «Подключить» к URL-адресу с вашим идентификатором клиента и URI переадресации, например:

https://acuityscheduling.com/oauth2/authorize?    response_type=code&scope=api-v1&client_id=YOUR_CLIENT_ID    &redirect_uri=YOUR_REDIRECT_URI

Люди будут отправлены на запрос авторизации клиента для своей учетной записи:

Вы также можете полагаться на библиотеки SDK Acuity для PHP и JavaScript, например:

<?phprequire_once("vendor/autoload.php");$acuity = new AcuitySchedulingOAuth(array(  "clientId" => "YOUR_CLIENT_ID",  "clientSecret" => "YOUR_CLIENT_SECRET",  "redirectUri" => "YOUR_REDIRECT_URI"));$acuity->authorizeRedirect(array("scope" => "api-v1"));

Acuity проверит ваши client_id, redirect_uri и scope, а затем отобразит экран входа пользователю, чтобы предоставить вашему приложению доступ к их учетной записи Acuity. Как только пользователь подключится, Acuity отправит их обратно в redirect_uri с authorization_code. Например, где example.org - URL вашего приложения:

https://example.org/oauth2?code=AN_AUTHORIZATION_CODE

В ответ на AUTHORIZATION_CODE вы выполните что-то вроде этого, чтобы запросить access_token:

curl https://acuityscheduling.com/oauth2/token \ -d grant_type=authorization_code \ -d code=AN_AUTHORIZATION_CODE \ -d redirect_uri= YOUR_REDIRECT_URI \ -d client_id=YOUR_CLIENT_ID \ -d client_secret=YOUR_CLIENT_SECRET

Или, в JavaScript:

var Acuity = require("acuityscheduling");var express = require("express");var url = require("url");var port = process.env.PORT || 8000;var app = express();app.get(url.parse("YOUR_REDIRECT_URI").path, function (req, res) {  var acuity = Acuity.oauth({    clientId: "YOUR_CLIENT_ID",    clientSecret: "YOUR_CLIENT_SECRET",    redirectUri: "YOUR_REDIRECT_URI"  });  acuity.requestAccessToken(req.query.code, function (err, tokenResponse) {    if (err) return res.send(err);    res.send(tokenResponse);  });});app.listen(port, function () {  console.log("Listening on %s", port);}).on("error", function () {  if (e) return console.error(e);});

Если все будет проверено, вы получите ответ в виде JSON, который включает access_token:

{  "access_token": "AN_ACCESS_TOKEN",  "token_type": "Bearer"}

Затем сохраните ACCESS_TOKEN для вашего пользователя, который будет использоваться для текущих запросов API.

Вот пример запроса от имени члена или учетной записи Acuity, используя приобретенный ACCESS_TOKEN:

<?phprequire_once("vendor/autoload.php");$acuity = new AcuitySchedulingOAuth(array(  "accessToken" => "AN_ACCESS_TOKEN"));$me = $acuity->request("me");print_r($me);

Для глубоко интегрированных сервисов Acuity обеспечивает почти мгновенное расширение вашей платформы и ее возможностей, а также простой способ расширить использование вашей компанией Acuity в других сервисах.

Как я уже писал, если вы это себе представляете, платформа разработчика Acuity в значительной степени обеспечивает всё это.

Веб-хуки Acuity

Acuity API также предоставляет веб-хуки для уведомления вашего приложения, когда назначение запланировано, отменено или перенесено. В меню Интеграция Acuity установите интеграцию Webhook с URL-адресом, который будет уведомлен об изменениях в назначениях (как показано выше). URL-адрес должен быть до порта 443 или 80.

Запросы Webhook представляют собой запросы POST application/x-www-form-urlencoded с этими переменными:

  • action: либо scheduled, rescheduled, canceled, либо changed, в зависимости от действия, инициировавшего вызов веб-хука.
  • id: идентификатор для встречи, получить информацию о назначении через вызов API получения назначения.
  • calendarID: идентификатор календаря для назначения.
  • appointmentTypeID: идентификатор типа назначения.

Различные действия:

  • scheduled вызывается один раз, когда назначение изначально забронировано.
  • rescheduled вызывается, когда назначение переносится на новое время.
  • canceled, вызывается всякий раз, когда назначение отменяется.
  • changed когда назначение изменено каким-либо образом. Это включает в себя, когда оно изначально запланировано, перенесено или отменено, а также когда информация о встрече, такая как адрес электронной почты или формы приема, обновляется.

Acuity выведет ваш API, и вы сможете перевести полученные данные для записи обновлений в новые и существующие встречи:

POST https://example.com/webhook-callback action=changed&id=13    &calendarID=1&appointmentTypeID=13

Вы также можете использовать curl, чтобы имитировать веб-хуки Acuity, чтобы помочь себе в разработке:

curl -d "action=changed&id=1&calendarID=1    &appointmentTypeID=1"     "https://example.com/webhook-callback"

Проверка запросов веб-хука

Уведомления Webhook подписываются Acuity с помощью вашего ключа API, поэтому вы можете проверить, что уведомление получено от Acuity. В документации по API есть богатая информация и примеры.

Отчеты о встречах

Одним из очевидных требований к представлению данных является указание назначений, поступающих в систему для различных членов команды от клиентов. API делает это простым.

Вот пример PHP SDK для запроса встреч из системы:

<?phprequire_once("vendor/autoload.php");$acuity = new AcuityScheduling(array(  "userId" => ACUITY_USER_ID,  "apiKey" => "ACUITY_API_KEY"));$appointments = $acuity->request("appointments");print_r($appointments);

Вот аналогичный curl запрос для календаря конкретного пользователя:

curl -u ACUITY_USER_ID:ACUITY_API_KEY "https://acuityscheduling.com/api/v1/appointments?max=25    &calendarID=27238"

Отчеты по заказам

Поскольку Acuity также позволяет вам перечислять и продавать продукты с помощью своих услуг по бронированию билетов, /orders возвращает список заказов из интернет-магазина с самыми последними в первую очередь.

Вот пример curl:

curl -u ACUITY_USER_ID:ACUITY_API_KEY     "https://acuityscheduling.com/api/v1/orders"

Хотя это не показано выше, вы можете запросить статус как delivered, paid или unpaid и запросить имена продуктов с использованием переменной заголовка в качестве списка разделенных запятой и списка упорядоченных имен продуктов.

Вот пример результатов:
