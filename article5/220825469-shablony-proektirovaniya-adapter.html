Шаблоны проектирования: Адаптер...

CodeDesign Patterns

Интро

В этой статье мы рассмотрели то, как шаблон проектирования фасад может быть использован для упрощения реализации любой большой и сложной системы, используя всего лишь один класс-фасад. ...

В этой статье мы рассмотрели то, как шаблон проектирования фасад может быть использован для упрощения реализации любой большой и сложной системы, используя всего лишь один класс-фасад.

В данной статье мы продолжим обсуждение шаблонов проектирования на примере шаблона адаптер. Применение этого шаблона поможет сделать ваш код независимым от внешних API или других любых классов, которые могут часто меняться. Этот шаблон попадает в категорию "структурных шаблонов", потому что показывает нам каким образом следует организовать и структурировать наши классы, чтобы затем можно было их легко использовать или расширять.

Опять же я хотел бы повторить, что шаблоны проектирования не вносят ничего нового в традиционные классы. Вместо этого они показывают нам правильные способы, как организовать наши классы, управлять их поведением и осуществлять создание объектов.

Проблема

<?phpclass PayPal {    public function __construct() {  // Your Code here //}public function sendPayment($amount) {// Paying via Paypal //echo "Paying via PayPal: ". $amount;}}$paypal = new PayPal();$paypal->sendPayment("2629");

В коде выше мы можете используем класс PayPal для оплаты определенной суммы. Здесь мы напрямую создаем объект класса PayPal и выполняем платеж через него. У вас есть этот код, разбросанный в нескольких местах. Можно заметить, что код использует для оплаты метод $paypal->sendPayment("amount here");.

Спустя некоторое время PayPal меняет имя метода из своего API с sendPayment на payAmount. Это должно четко указывать на проблему для тех из нас, кто использовал метод sendPayment. В частности нам нужно теперь изменить все вызовы метода sendPayment на payAmount. Представьте себе количество кода, которое нам нужно изменить и время, которе потребуется потратить на тестирование функционала еще раз.

Решение

Одним из решений этой проблемы является использование шаблона проектирования адаптер.

Согласно Википедии:

В области программного обеспечения шаблон адаптер является шаблоном проектирования программного обеспечения, который позволяет использовать интерфейс существующего класса из другого интерфейса. Он часто используется чтобы дать возможность существующим классам работать с другими классами без изменения их исходного кода.

В этом случае мы должны создать интерфейс оболочки, который сделает это возможным. Мы не будем делать каких либо изменений в библиотеке с внешним классом, потому что мы не имеем контроля над ним, и он может быть изменен в любое время.

Давайте углубимся в код, который показывает шаблон адаптер в действии:

// Concrete Implementation of PayPal Classclass PayPal {    public function __construct() {// Your Code here //}public function sendPayment($amount) {// Paying via Paypal //echo "Paying via PayPal: ". $amount;}}// Simple Interface for each Adapter we createinterface paymentAdapter {    public function pay($amount);}class paypalAdapter implements paymentAdapter {    private $paypal;public function __construct(PayPal $paypal) {$this->paypal = $paypal;}public function pay($amount) {$this->paypal->sendPayment($amount);}}

Изучая приведенный выше код, вы должны были заметить, что мы не сделали никаких изменений в основном классе PayPal. Вместо этого мы создали единый интерфейс для нашего платежного адаптера и класс адаптера для PayPal.

Затем мы создали объект класса адаптера вместо основного класса PayPal. При создании объекта класса адаптера мы будем передавать объект основного класса PayPal как аргумент, так что класс адаптера будет иметь ссылку на основной класс, и он может вызывать необходимые методы класса PayPal.

Давайте выясним, как мы можем использовать этот способ напрямую:

// Client Code$paypal = new paypalAdapter(new PayPal());$paypal->pay("2629");

Теперь представьте что PayPal меняет имя метода sendPayment на payAmount. Теперь нам просто нужно внести изменения в paypalAdapter. Просто взгляните на код обновленного адаптера, который имеет только одно изменение.

class paypalAdapter implements paymentAdapter {    private $paypal;public function __construct(PayPal $paypal) {$this->paypal = $paypal;}public function pay($amount) {$this->paypal->payAmount($amount);}}

Всего одно лишь изменение и все.

Добавление Нового Адаптера

На данный момент мы видели, как мы можем использовать шаблон проектирования адаптер для преодоления вышеуказанных проблем. Теперь очень легко добавить новый класс, основываясь на существующем адаптере. Допустим у нас есть MoneyBooker API для оплаты.

Затем вместо непосредственного использования класса MoneyBooker, мы должны применить тот же шаблон адаптер, который мы использовали для PayPal.

// Concrete Implementation of MoneyBooker Classclass MoneyBooker {    public function __construct() {// Your Code here //}public function doPayment($amount) {// Paying via MoneyBooker //echo "Paying via MoneyBooker: ".  $amount;}}// MoneyBooker Adapterclass moneybookerAdapter implements paymentAdapter {    private $moneybooker;public function __construct(MoneyBooker $moneybooker) {$this->moneybooker = $moneybooker;}public function pay($amount) {$this->moneybooker->doPayment($amount);}}// Client Code$moneybooker = new moneybookerAdapter(new MoneyBooker());$moneybooker->pay("2629");

Как вы можете видеть, применяются те же самые принципы. Определите метод, который доступен для третьих классов и затем, если какой-либо из них изменит свое API, вы просто меняете зависимый класс, не показывая его внешний интерфейс.

Заключение

Большое приложение постоянно использует другие библиотеки и API, поэтому я хотел бы предложить использовать шаблон адаптер, чтобы наша кодовая база не зависела от изменений третьих библиотек или API.

Я сделал все возможное для объяснения основ и полезных примеров шаблона проектирования адаптер, но если у вас есть какие-либо вопросы или комментарии, буду рад иметь от вас обратную связь.
