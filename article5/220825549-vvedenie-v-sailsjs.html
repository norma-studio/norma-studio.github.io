Введение в Sails.js...

CodeWeb Development

Интро

Sails-- это фреймворк Javascript, разработанный так, чтобы напоминать архитектуру MVC из фреймворков, таких как Ruby on Rails. Это упрощает процесс создания приложений Node.js, особенно API, одностраничных приложений и функций реального времени, таких как чат....

Sails - это фреймворк Javascript, разработанный так, чтобы напоминать архитектуру MVC из фреймворков, таких как Ruby on Rails. Это упрощает процесс создания приложений Node.js, особенно API, одностраничных приложений и функций реального времени, таких как чат.

Установка

Установить Sails довольно просто. Для установки нам нужен будет Node.js, а также npm, который поставляется с node. Затем в терминале необходимо выполнить следующую команду:

sudo npm install sails -g

Создаем новый проект

Чтобы создать новый проект Sails, используется следующая команда:

sails new myNewProject

Sails создаст новую папку с именем myNewProject и добавит все необходимые файлы для создания базового приложения. Чтобы узнать, что было создано, просто войдите в папку myNewProject и запустите сервер Sails, выполнив следующую команду в терминале:

sails lift

Порт Sails по умолчанию - 1337, поэтому, если вы откроете http://localhost:1337, вы должны получить страницу index.html .

Теперь давайте посмотрим, что для нас сгенерировал Sails . В нашей папке myNewProject были созданы следующие файлы и подпапки:

Папка assets

Папка «assets» содержит подкаталоги для файлов Javascript и CSS, которые должны быть загружены во время выполнения. Это лучшее место для хранения вспомогательных библиотек, используемых вашим приложением.

Папка public

Содержит общедоступные файлы, такие как изображения, используемые вашим сайтом, значки и т.д.

Папка config

Это одна из важных папок. Sails рассчитан на гибкость. Он предполагает некоторые стандартные соглашения, но также позволяет разработчику изменять способ, которым Sails настраивает созданное приложение в соответствии с потребностями проекта. Ниже приведен список конфигурационных файлов, присутствующих в папке config:

  • adapters.js - используется для настройки адаптеров базы данных
  • application.js - общие настройки для приложения
  • assets.js - настройки assets для CSS и JS
  • bootstrap.js - код, который будет запущен до запуска приложения
  • locales - папка, содержащая переводы
  • policy.js - настройка управления правами пользователя
  • routes.js - маршруты для системы
  • views.js - просмотр соответствующих настроек

Документация sails.js содержит подробную информацию о каждой из этих папок.

Папка views

Представления приложения хранятся в этой папке. Рассматривая ее содержимое, мы замечаем, что представления генерируются по умолчанию как EJS (встроенный JavaScript). Кроме того, папка views содержит представления для обработки ошибок (404 и 500), а также файл макета (layout.ejs) и представления для контроллера home, которые были созданы Sails.

Папка api

Эта папка состоит из группы подпапок:

  • папка adapters содержит адаптеры, используемые приложением для
    обработки подключения к базе данных
  • папка controllers содержит контроллеры приложения
  • модели приложения хранятся в папке models
  • в папке policies хранятся правила доступа пользователей приложения
  • службы api, реализованные приложением, хранятся в services
    папка

Настройка приложения

Пока мы создали наше приложение и посмотрели, что было создано по умолчанию, теперь пришло время настроить приложение, чтобы оно соответствовало нашим потребностям.

Общие настройки

Общие настройки хранятся в файле config/application.js. Конфигурируемые параметры для приложения:

  • имя приложения (appName)
  • порт, на котором приложение будет прослушивать (port)
  • среда приложения; может быть либо development, либо production (environment)
  • уровень для логгера, используемый для управления размером файла лога (log)

Обратите внимание, что, устанавливая environment приложения как production, пакет Sails связывает и минимизирует CSS и JS, что затрудняет отладку.

Маршруты

Маршруты приложений определяются в файле config/routes.js. Как и следовало ожидать, вы будете чаще всего работать с этим файлом, поскольку вы будете добавлять новые контроллеры в приложение.

Маршруты экспортируются следующим образом в файле конфигурации:

module.exports.routes = {  // route to index page of the home controller  "/": {    controller: "home"  },  // route to the auth controller, login action  "/login": {    controller: "auth",    action: "login"  },  // route to blog controller, add action to add a post to a blog  // note that we use also the HTTP method/verb before the path  "post /blog/add": {    controller: "blog",    action: "add_post"  },  // route to get the first blog post. The find action will return  // the database row containing the desired information  "/blog/:item": {    controller: blog,    action: find  }}

Views

Что касается представлений, настраиваемые параметры - это механизм шаблонов, который будет использоваться.

Models

Модели представляют собой данные приложения, хранящиеся в базе данных. Модели определяются с помощью атрибутов и ассоциаций. Например, определение модели Person может выглядеть так:

// Person.jsvar Person = {  name: "STRING",  age: "INTEGER",  birthDate: "DATE",  phoneNumber: "STRING",  emailAddress: "STRING"};exports = Person;

Связь с базой данных осуществляется через адаптеры. Адаптеры определены в api/adapters и настроены в файле adapters.js. На момент написания этой статьи Sails поставляется с тремя адаптерами: memory, disk и mysql, но вы можете написать свой собственный адаптер (подробности см. в документации).

После определения модели вы можете работать с ней, создавая записи, находить записи, обновлять и удалять записи.

Контроллеры

Контроллеры размещаются в api/controllers. Контроллер создается с помощью следующей команды:

sails generate controller comment

Эта команда создаст объект CommentController. Действия определяются внутри этого объекта. Действия также могут быть сгенерированы командой generate controller:

sails generate controller comment create destroy tag like

Это создаст контроллер Comment с действиями для create, destroy, tag и like.

Действия получают в качестве параметров запрос и объекты ответа, которые могут использоваться для получения параметров URI (объекта запроса) или вывода в представлении (с использованием объекта ответа).

Для связи с моделью используется обратный вызов соответствующего действия. Например, в случае запроса базы данных с помощью find для управления моделью используется следующий шаблон:

Blog.find(id).done(err, blog) {  // blog is the database record with the specified id  console.log(blog.content);}

Views

Представления используются для обработки пользовательского интерфейса приложения. По умолчанию представления обрабатываются с помощью EJS, но можно использовать любую другую библиотеку шаблонов. Как настраивать представления обсуждалось ранее в главе «Конфигурация».

Представления определяются в каталоге /views, а шаблоны определены в папке /assests/templates.

Существует, в основном, четыре типа представлений:

  • server-side представления
  • частичные представления
  • лэйауты
  • client-side представления

Server-Side представления

Их задача - отображать данные, когда клиент запрашивает представление. Обычно метод res.view соответствует клиенту с соответствующим представлением. Но если для запроса не существует контроллера или действия, Sails будет обслуживать представление следующим образом: /views/:controller/:action.ejs.

Представление макета

Макет можно найти в /views/layout.ejs. Он используется для загрузки assets, таких как таблицы стилей или библиотеки JavaScript.

Посмотрите на указанный файл:

<!DOCTYPE html><html>  <head>    <title><%- title %></title>    <!-- Viewport mobile tag for sensible mobile support -->    <meta name=&quot;viewport&quot; content=&quot;width=device-width, initial-scale=1, maximum-scale=1&quot;>    <!-- JavaScript and stylesheets from your public folder are included here -->    <%- assets.css() %>    <%- assets.js() %>  </head>  <body>    <%- body %>    <!-- Templates from your view path are included here -->    <%- assets.templateLibrary() %>  </body></html>

Строки assets.css() и assets.js() загружают ресурсы CSS и JS вашего приложения, а assets.templateLibrary загрузит клиентские шаблоны.

Шаблоны на стороне клиента

Они определены в /assets/templates и загружаются, как мы видели выше.

Маршруты

Мы обсудили, как настроить маршруты в главе «Конфигурация».

Существует несколько соглашений, которым следует Sails, когда обрабатываются маршруты:

  • если URL-адрес не указан в config/routes.js, маршрутом по умолчанию для URL является /:controller/:action/:id с очевидными значениями для контроллера и действия, а id - параметр запроса, полученный из URL-адреса.
  • если :action не указано, Sails перенаправляет на соответствующее действие. Из коробки используются те же правила маршрута RESTful, что и в Backbone.
  • если запрошенный контроллер / действие не существует, Sails будет вести себя так:
    • если существует представление, Sails отобразит это представление
    • если представление не существует, но существует модель, Sails вернет форму JSON этой модели
    • если ни один из вышеперечисленных не существует, Sails ответит 404
