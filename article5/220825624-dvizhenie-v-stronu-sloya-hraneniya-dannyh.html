Потребность приложения сохранять свое состояние и данные является настолько значимой, что существуют десятки, если не сотни различных технологий для решения этой единственной проблемы. Каждое приложение, а иногда и каждый отдельный компонент приложения сам по себе уникален и поэтому нуждается в своем уникальном решении.

В этой статье я покажу лучшие практики, чтобы помочь вам определиться, какой подход выбрать при работе над будущими приложениями. Я вкратце рассмотрю некоторые шаблоны высокого уровня, а затем более подробно остановлюсь на шаблоне Active Record, и несколько слов добавлю о шаблоне проектирования Шлюз.

Конечно я не научу вас всей теории о проектировании, но я покажу вам несколько примеров, которые начинаются как неорганизованный код, который затем трансформируется в структурированное правильное решение.

Две Истории Одного Приложения

База данных существует для данных, а не для кода.

Сегодня редкий программист может понять эту архаичную систему.

Старый проект, над которым мне пришлось работать, был начат в 2000 году. Тогда команда программистов начала новый проект с оценки различных требований, размышляя о рабочей нагрузке приложения. Были испытаны различные технологии и команда пришла к следующему выводу: весь PHP-код приложения, за исключением файла index.php, должен находиться в базе данных MySQL. Их решение может показаться возмутительным сегодня, но это было приемлемым, двенадцать лет назад (OK... может быть не было).

Они начали с создания основных таблиц, а затем и других для каждой веб-страницы в отдельности. Такое решение работало... некоторе время. Первоначальные авторы знали, как поддерживать код, но потом каждый автор уходил один за другим, оставляя кодовую базу в руках новичков.

Сегодня ни один программист уже не сможет понять этой архаичной системы. Все начинается с запроса MySQL из index.php. В результате этого запроса возвращается некоторый PHP-код, который выполняет другие запросы. Простейший сценарий включает в себя по меньшей мере пять таблиц базы данных. Естественно нет никаких тестов или спецификаций. Пытаться что-нибудь изменить бесполезно, и мы просто должны были переписывать весь модуль, если что-то в нем работало не так.

Изначальные разработчики проигнорировали тот факт, что база данных должна содержать только данные, а не бизнес-логику. Они смешали MySQL с PHP и HTML и проигнорировали концепции проектирования высокого уровня.

HTML Является Только Средством Представления

Все приложения должны сосредоточиться на соблюдении проектирования высокого уровня.

Шло время, и новым программистам нужно было добавлять новый функционал к системе, в то же время, исправляя старые баги. Не было никакой возможности продолжать использование таблиц MySQL для всего, и каждый кто участвовал в поддержке согласились с тем, что архитектура имеет ужасные недостатки. Поэтому новые программисты оценили различные требования, думая о рабочих нагрузках, которые приложению придется обрабатывать, испытали различные технологии и пришли к выводу: вынести из таблиц столько кода, сколько успеют к финальной презентации. Опять же это решение может показаться возмутительным сегодня, но это были светлые времена той ужасной архитектуры.

Разработчики взяли за основу шаблонный фреймворк и выстроили приложение вокруг него, начиная с нового шаблона каждый добавляемый функционал. Это было легко; шаблон является описательным, и они знали, где найти код, который выполняет определенную задачу. Но в конечном итоге они пришли к файлам шаблонов, которые содержали Domain Specific Language (DSL), HTML, PHP и конечно же MySQL запросы.

Сегодня моя команда просто смотрит и удивляется. Это просто чудо, что многие из отображений действительно работали. Может занять огромное количество времени, чтобы определить, каким образом информация поступает из базы данных в представления. Как и было до этого кодовая база все еще представляет собой один большой беспорядок!

Те разработчики проигнорировали тот факт, что представления не должны содержать бизнес-логику. Они смешивали MySQL с PHP и HTML и игнорировали концепции архитектуры высокого уровня.

Архитектура Приложения Высокого Уровня

Mock — это объект, который действует как его настоящий двойник, но не выполняет реальный код.

Все приложения должны соблюдать чистую архитектуру верхнего уровня. Это не всегда достижимо, но должно быть основным приоритетом. Правильно построенная архитектура всегда содержит хорошо изолированную бизнес-логику. Создание, сохранение и перемещение объектов вынесены из ядра приложения, а зависимости указывают только на бизнес-логику.

Изолируя бизнес логику, вы открываете дверь для огромных возможностей все становится в своем роде плагинами, если внешние зависимости всегда указывают на бизнес-логику. Например, вы можете переключиться с тяжелой базы данных MySQL на легковесную базу SQLite3.

  • Представьте, что можете легко отказаться от вашего MVC фреймворка и заменить его другим, не касаясь при этом вашей бизнес-логики.
  • Представьте, что вы теперь предоставляете результаты работы вашего приложения через стороннее API, а не посредством HTTP запросов, или заменяете любую стороннюю технологию, которую сейчас используете, не затрагивая бизнес-логики (кроме языка программирования конечно же).
  • Представьте себе все эти изменения, а ваши тесты до сих пор успешно проходят.

Реализация Рабочего Решения для Хранения Поста в Блоге

Чтобы лучше понять всю суть проблемы с плохой, слабо организованной архитектурой, я начну, как вы уже догадались, с простого примера с блогом. На протяжении этой статьи я буду следовать некоторым принципам разработки через тестирование (TDD) и сделаю мои тесты легко читаемыми - даже для тех, кто не имеет опыта разработки с использованием TDD. Представим, что вы используете MVC фреймворк. При сохранении поста в блоге, контроллер BlogPost выполняет метод save(). Этот метод соединяется с базой данных SQLite, чтобы сохранить пост блога в базе.

Давайте создадим директорию в нашем каталоге с кодом и назовем ее Data, затем просмотрим ее через консоль. Создадим базу данных и таблицу следующим образом:

$ sqlite3 MyBlogSQLite version 3.7.13 2012-06-11 02:05:22Enter ".help" for instructionsEnter SQL statements terminated with a ";"sqlite> create table BlogPosts (  title varchar(120) primary key,content text,published_timestamp timestamp);

Наш метод save() получает значения из формы в виде массива $data:

class BlogPostController {function save($data) {$dbhandle = new SQLite3("Data/MyBlog");$query = "INSERT INTO BlogPosts VALUES("" . $data["title"] . "","" . $data["content"] . "","" . time(). "")";$dbhandle->exec($query);}}

Этот код работает, это можно проверить, вызвав его из другого класса и передав заранее заполненный данными массив $data:

$this->object = new BlogPostController;$data["title"] = "First Post Title";$data["content"] = "Some cool content for the first post";$data["published_timestamp"] = time();$this->object->save($data);

Содержимое переменной $data действительно сохранилось в базе данных:

sqlite> select * from BlogPosts;First Post Title|Some cool content for the first post|1345665216

Описательные Тесты

Наследование является сильнейшим типом зависимости.

Описательный тест проверяет текущее поведение предварительно написанного кода. Такие тесты чаще всего используются для описания legacy кода, и делают рефакторинг такого кода гораздо проще.

Описательный текст может тестировать модуль, компонент или пройти целый путь от UI к базе данных; все зависит от того, что мы хотим протестировать. В нашем случае тест выполнит контроллер и проверит содержимое базы данных. Это не типичный юнит, функциональный или интеграционный тест, и обычно такие тесты не могут быть четко ассоциированы с этими уровнями тестирования.

Описательные тесты - это временная мера безопасности, и мы обычно удаляем их после того, как код будет отрефакторен и протестирован юнит тестами. Вот реализация теста, который находится в директории Test:

require_once "../BlogPostController.php";class BlogPostControllerTest extends PHPUnit_Framework_TestCase {private $object;private $dbhandle;function setUp() {$this->object = new BlogPostController;$this->dbhandle = new SQLite3("../Data/MyBlog");}function testSave() {$this->cleanUPDatabase();$data["title"] = "First Post Title";$data["content"] = "Some cool content for the first post";$data["published_timestamp"] = time();$this->object->save($data);$this->assertEquals($data, $this->getPostsFromDB());}private function cleanUPDatabase() {$this->dbhandle->exec("DELETE FROM BlogPosts");}private function getPostsFromDB() {$result = $this->dbhandle->query("SELECT * FROM BlogPosts");return $result->fetchArray(SQLITE3_ASSOC);}}

Этот тест создает новый объект контроллера и выполняет метод save(): Затем тест читает информацию из базы данных и сравнивает с предварительно определенным массивом $data[]; Мы выполняем это сравнение используя метод $this->assertEquals(), который проверяет, что оба его параметры одинаковы. Если они различаются, то тест не пройдет. Кроме того мы очищаем таблицу BlogPosts каждый раз, когда мы запускаем тест.

Устаревший код является нетестируемым. -Майкл Фитерз

Запуская наши тесты, давайте попробуем очистить немного кода. Откройте базу данных с именем каталога и воспользуемся функции sprintf() чтобы составить строку запроса. Это приведет к гораздо более простому код:

class BlogPostController {function save($data) {$dbhandle = new SQLite3(__DIR__ . "/Data/MyBlog");$query = sprintf("INSERT INTO BlogPosts VALUES ("%s","%s","%s")", $data["title"], $data["content"], time());$dbhandle->exec($query);}}

Шаблон Шлюз к Данным Таблицы

Мы признаем, что наш код необходимо перенести из контроллера в бизнес-логику и в слой хранением, а шаблон шлюза может помочь нам начать работу на этом пути. Вот пересмотренный testSave() метод:

function testItCanPersistABlogPost() {$data = array("title" => "First Post Title", "content" => "Some content.", "timestamp" => time());$blogPost = new BlogPost($data["title"], $data["content"], $data["timestamp"]);$mockedPersistence = $this->getMock("SqlitePost");$mockedPersistence->expects($this->once())->method("persist")->with($blogPost);$controller = new BlogPostController($mockedPersistence);$controller->save($data);}

Вот так нам следует использовать метод save() в контроллере. Нам нужно в контроллере вызывать метод persist($blogPostObject) объекта шлюза. Давайте внесем эти изменения в BlogPostController:

class BlogPostController {private $gateway;function __construct(Gateway $gateway = null) {$this->gateway = $gateway ? : new SqlitePost();}function save($data) {$this->gateway->persist(new BlogPost($data["title"], $data["content"], $data["timestamp"]));}}

Хорошо спроектированная архитектура имеет хорошо изолированную бизнес-логику.

Отлично! Наш класс BlogPostController стал гораздо проще. Он использует шлюз для хранения данных, вызывая метод persist(). Контроллеру нет необходимости знать, каким образом хранятся данные и логика становится модульной.

В предыдущем примере мы создавали контроллер с заглушкой объекта, который отвечает за хранение данных, чтобы быть уверенными, что данные никогда не будут записаны во время тестов. В продакшене контроллер создает свой собственный объект для сохранения данных, используя объект SqlitePost. Заглушка - это объект, который ведет себя так же как его реальный двойник, но при этом никакой код не выполняется.

Теперь давайте попробуем достать пост блога из хранилища данных. Это так же просто, как и сохранение данных, но пожалуйста, обратите внимание, что я немного переделал тест.

require_once "../BlogPostController.php";require_once "../BlogPost.php";require_once "../SqlitePost.php";class BlogPostControllerTest extends PHPUnit_Framework_TestCase {private $mockedPersistence;private $controller;private $data;function setUp() {$this->mockedPersistence = $this->getMock("SqlitePost");$this->controller = new BlogPostController($this->mockedPersistence);$this->data = array("title" => "First Post Title", "content" => "Some content.", "timestamp" => time());}function testItCanPersistABlogPost() {$blogPost = $this->aBlogPost();$this->mockedPersistence->expects($this->once())->method("persist")->with($blogPost);$this->controller->save($this->data);}function testItCanRetrievABlogPostByTitle() {$expectedBlogpost = $this->aBlogPost();$this->mockedPersistence->expects($this->once())->method("findByTitle")->with($this->data["title"])->will($this->returnValue($expectedBlogpost));$this->assertEquals($expectedBlogpost, $this->controller->findByTitle($this->data["title"]));}public function aBlogPost() {return new BlogPost($this->data["title"], $this->data["content"], $this->data["timestamp"]);}}

И реализация в BlogPostController заключена в методе из всего одного выражения:

function findByTitle($title) {return $this->gateway->findByTitle($title);}

Круто не так ли? Класс BlogPost теперь является частью бизнес-логики (помните схему архитектуры, описанную выше). UI/MVC создает объект класса BlogPost и использует конкретную реализацию Gateway, чтобы сохранить данные. Все зависимости указывают на бизнес-логику.

Остался всего один шаг: создать конкретную реализацию Gateway. Ниже приведен класс SqlitePost:

require_once "Gateway.php";class SqlitePost implements Gateway {private $dbhandle;function __construct($dbhandle = null) {$this->dbhandle = $dbhandle ? : new SQLite3(__DIR__ . "/Data/MyBlog");}public function persist(BlogPost $blogPost) {$query = sprintf("INSERT INTO BlogPosts VALUES ("%s","%s","%s")", $blogPost->title, $blogPost->content, $blogPost->timestamp);$this->dbhandle->exec($query);}public function findByTitle($title) {$SqliteResult = $this->dbhandle->query(sprintf("SELECT * FROM BlogPosts WHERE title = "%s"", $title));$blogPostAsString = $SqliteResult->fetchArray(SQLITE3_ASSOC);return new BlogPost($blogPostAsString["title"], $blogPostAsString["content"], $blogPostAsString["timestamp"]);}}

Примечание: Тест для этой реализации также доступен в исходном коде, но из-за его сложности и длины, я не стал его здесь включать.

Active Record является одним из самых спорных шаблонов. Некоторые его все же активно используют (например Rails и CakePHP), другие наоборот избегают. Множество приложений Object Relational Mapping (ORM) используют этот шаблон для сохранения объектов в таблицы. Вот его схема:

Как вы можете видеть, объекты, основанные на Active Record могут сами себя сохранять и извлекать. Обычно это достигается наследованием от базового класса ActiveRecordBase, это класс который знает как работать с базой данных.

Продолжая Тему Active Record

Есть много других шаблонов, которые тесно связаны с сохранением, например шаблон Прокси.

Если бы мы следовали шаблону Active Record, в точности как он представлен Мартин Фаулером в своей книге 2003 года, Шаблоны Корпоративных Приложений, то нам придется переместить SQL-запросы в класс BlogPost. Это, однако, повлечет за собой нарушения Принципа Инверсии Зависимостей и Принципа Открытости-Закрытости. Принцип Инверсии Зависимостей гласит:

  • Модули высокого уровня не должны зависеть от модулей низкого уровня. Оба должны зависеть от абстракций.
  • Абстракции также не должны зависеть от деталей. Детали должны зависеть от абстракций.

А Принцип Открытости-Закрытости утверждает о том, что сущности (классы, модули, функции и т.д.) должны быть открыты для расширения, но закрыты для модификации. Мы будем использовать более интересный подход и интегрируем шлюз в наше решение с Active Record.

Если вы попробовали сделать это сами, то уже наверное поняли, что добавление шаблона Active Record запутает код. По этой причине я решил отключить тесты контроллера и SqlitePost и сосредоточиться только на классе BlogPost. Первые шаги: заставить BlogPost загружать себя самого, сделав его конструктор приватным и подключить его к интерфейсу шлюза. Вот первая версия файла BlogPostTest:

require_once "../BlogPost.php";require_once "../InMemoryPost.php";require_once "../ActiveRecordBase.php";class BlogPostTest extends PHPUnit_Framework_TestCase {function testItCanConnectPostToGateway() {$blogPost = BlogPost::load();$blogPost->setGateway($this->inMemoryPost());$this->assertEquals($blogPost->getGateway(), $this->inMemoryPost());}function testItCanCreateANewAndEmptyBlogPost() {$blogPost = BlogPost::load();$this->assertNull($blogPost->title);$this->assertNull($blogPost->content);$this->assertNull($blogPost->timestamp);$this->assertInstanceOf("Gateway", $blogPost->getGateway());}private function inMemoryPost() {return new InMemoryPost();}}

Он проверяет, что в пост в блоге инициализируется правильно и что он может иметь шлюз. Это хорошая практика, использовать несколько ассертов, когда они все тестируют одну и ту же концепцию и логику.

Наш второй тест тоже имеет несколько ассертов, но все они направлены на общую концепцию пустого поста в блоге. Конечно класс BlogPost так же следует модифицировать:

class BlogPost {private $title;private $content;private $timestamp;private static $gateway;private function __construct($title = null, $content = null, $timestamp = null) {$this->title = $title;$this->content = $content;$this->timestamp = $timestamp;}function __get($name) {return $this->$name;}function setGateway($gateway) {self::$gateway = $gateway;}function getGateway() {return self::$gateway;}static function load() {if(!self::$gateway) self::$gateway = new SqlitePost();return new self;}}

Теперь у него есть метод load(), который возвращает новый объект в правильным шлюзом. Исходя из этого мы продолжим реализацию метода load($title) для создания нового BlogPost с информацией из базы данных. Для более легкого тестирования я создам класс InMemoryPost для хранения. Он всего лишь будет хранить в памяти список объектов и возвращать их по запросу:

class InMemoryPost implements Gateway {private $blogPosts = array();public function findByTitle($blogPostTitle) {return array("title" => $this->blogPosts[$blogPostTitle]->title,"content" => $this->blogPosts[$blogPostTitle]->content,"timestamp" => $this->blogPosts[$blogPostTitle]->timestamp);}public function persist(BlogPost $blogPostObject) {$this->blogPosts[$blogPostObject->title] = $blogPostObject;}}

Как вы можете видеть, я радикально изменил то, как теперь используется класс BlogPost.

class BlogPost {private $title;private $content;private $timestamp;private function __construct($title = null, $content = null, $timestamp = null) {$this->title = $title;$this->content = $content;$this->timestamp = $timestamp;}function __get($name) {return $this->$name;}static function load($title = null, $content = null, $timestamp = null, $gateway = null) {$gateway = $gateway ? : new SqlitePost();if(!$content) {$postArray = $gateway->findByTitle($title);if ($postArray) return new self($postArray["title"], $postArray["content"], $postArray["timestamp"]);}return new self($title, $content, $timestamp);}}

Метод load() проверяет параметр $content и создает новый экземпляр BlogPost, если значение параметра верное. Если нет, то метод пытается найти в блоге пост с заданным названием. Если пост был найден, то он будет возвращен, если нет - метод создаст пустой объект BlogPost.

Чтобы заставить этот код работать, нам также нужно будет изменить то, каким образом работает шлюз. Наша реализация должна возвращать ассоциативный массив с элементами title, content, и timestamp вместо самого объекта. Это соглашение, которое я выбрал.
