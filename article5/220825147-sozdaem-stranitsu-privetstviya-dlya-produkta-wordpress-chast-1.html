В первых двух статьях серии, я описал, что такое страницы приветствия, и как они улучшают опыт пользователей, связывая разные стороны продукта, далее я описал, интерфейс транзитного кэширования, который мы будем использовать при создании страницы приветствия.

Написание страницы приветствия для плагина WordPress может быть сложным процессом. Вся концепция построена на переадресации пользователей на определенную страницу через транзитное кэширование и последующее удаление его. Давайте начнем создавать страницу приветствия.

Шаблон страницы приветствия WordPress

Я намереваюсь создать шаблон страницы приветствия WordPress в виде плагина, который может быть использован с продуктом WordPress. Окончательный вид этого плагина расположен на GitHub в папке проекта WP-Welcome-Page-Boilerplate-For-TutsPlus.

Я предполагаю, что, на данный момент, у вас все настроено следующим образом:

  • локальный компьютер или веб сервер
  • консоль демо сайта WordPress

Архитектура плагина

Я создам простой плагин, который отобразит страницу приветствия, после того как пользователь установит и активирует плагин. Давайте рассмотрим архитектуру плагина:

  • WP-Welcome-Page-Boilerplate: папка, содержащая наш плагин.
  • WP-Welcome-Page-Boilerplate/welcome: папка, содержащая все, относящееся к странице приветствия.
  • WP-Welcome-Page-Boilerplate/welcome/img: папка с изображениями
  • wp-welcome-page-boilerplate.php: основной файл плагина в корневой папке плагина. Этот файл несет ответственность за определение констант, и включение файлов инициализаторов, как например, welcome-init.php.
  • welcome-init.php: файл инициализатор, который несет ответственность за три вещи: добавление временной переменной после активации плагина, удаление ее при деактивации плагина, и наконец добавление логического файла, например, welcome-logic.php
  • welcome-logic.php: логический файл, который несет ответственность за безопасную переадресацию страницы приветствия, создание подменю страницы приветствия, и добавление файла просмотра страницы приветствия, например, welcome-view.php
  • welcome-view.php: файл просмотра, который ответственен за отображение страницы приветствия, и написан на PHP и HTML.
  • Есть дополнительные папки для изображений и CSS, и они названы в соответствии с их функциями.

Можно просмотреть данный скриншот для проверки архитектуры.

Рабочий процесс плагина

Плагин работает следующим образом:

  • Добавляет транзитное кэширование при активации плагина.
  • Удаляет транзитное кэширование при деактивации плагина.
  • Безопасная переадресация на страницу приветствия.

Основной файл стандартного плагина

Давайте начнем наше обсуждение с содержания основного файла: wp-welcome-page-boilerplate.php. Вот полный код: 

<?php/** * Plugin Name: WP Welcome Page Boilerplate * Plugin URI: https://code.tutsplus.com/articles/building-a-welcome-page-for-your-wordpress-product-introduction--cms-26013 * Description: Welcome page boilerplate for WordPress plugins. * Version: 1.0.0 * Author: mrahmadawais, WPTie * Author URI: http://AhmadAwais.com/ * License: GPL-3.0+ * License URI: http://www.gnu.org/licenses/gpl-3.0.html * Domain Path: /lang * Text Domain: WPW */if ( ! defined( "WPINC" ) ) {    die;}

Раздел заголовка плагина PHPDoc дает знать WordPress, что файл представляет собой плагин. Установленные параметры дают знать, как обрабатываются данные. Заголовок может содержать, как минимум, название плагина, но некоторые моменты могут, и обычно должны, быть включены. О требованиях к заголовкам плагинов можете прочитать в руководстве по созданию плагинов WordPress.

Дальнейший раздел кода производит проверку безопасности. Если кто-то попытается получить прямой доступ к файлу, код запускает проверку ABSPATH, которая прерывает выполнение скрипта, если доступ производится извне.

После чего, код выглядит следующим образом:

// Plugin version.if ( ! defined( "WPW_VERSION" ) ) {    define( "WPW_VERSION", "1.0.0" );}// Plugin folder name.if ( ! defined( "WPW_NAME" ) ) {    define( "WPW_NAME", trim( dirname( plugin_basename( __FILE__ ) ), "/" ) );}// Plugin directory, including the folder.if ( ! defined( "WPW_DIR" ) ) {    define( "WPW_DIR", WP_PLUGIN_DIR . "/" . WPW_NAME );}// Plugin url, including the folder.if ( ! defined( "WPW_URL" ) ) {    define( "WPW_URL", WP_PLUGIN_URL . "/" . WPW_NAME );}// Plugin root file.if ( ! defined( "WPW_PLUGIN_FILE" ) ) {    define( "WPW_PLUGIN_FILE", __FILE__ );}

Я установил несколько глобальных констант, которые определяют следующие параметры: версию плагина, корневую папку, URL, и имя основного файла плагина.

Каждая из констант заключена в конструкцию if( ! defined() ), что помогает избежать ошибок при переопределении глобальных постоянных. Рекомендую устанавливать глобальные константы с префиксом имени программы. Название программы данного плагина - WPW, а именно, WordPress Welcome. Таким образом, каждая константа имеет префикс WPW_.

Установленными константами плагина являются:

  • WPW_VERSION: версия плагина
  • WPW_NAME: название папки плагина
  • WPW_DIR: корневая папка плагина
  • WPW_URL: URL плагина
  • WPW_PLUGIN_FILE: корневой файл плагина

Как только все константы определены, мы начнем писать содержание файла страницы приветствия.

И наконец, добавляем инициализатор приветствия и логику приветствия.

if ( file_exists( WPW_DIR . "/welcome/welcome-init.php" ) ) {    require_once( WPW_DIR . "/welcome/welcome-init.php" );}if ( file_exists( WPW_DIR . "/welcome/welcome-logic.php" ) ) {    require_once( WPW_DIR . "/welcome/welcome-logic.php" );}

Мне нравиться, чтобы код был минимальным, точным и хорошо документированным. Таким образом, вместо того чтобы добавлять каждый блок кода в один файл, я предпочитаю создавать отдельные файлы, каждый с одной единственной целью. Вот почему у нас есть файл инициализатор, который инициализирует все связанное со страницей приветствия.

Функция require_once() добавляет файл welcome-init.php, но перед этим, я всегда проверяю существует ли файл с помощью PHP функции file_exists(), чтобы избежать критических ошибок, если файл удален.

Те же принципы применимы к файлу welcome-logic.php, с которым мы будем работать в следующей части серии.

Можете просмотреть конечный код в файле wp-welcome-page-boilerplate.php на GitHub.

Инициализатор приветствия

Чтобы содержать в порядке все файлы, относящиеся к странице приветствия, я создал отдельную папку под названием welcome и добавил файл welcome-init.php. Весь код этого файла контролирует инициализацию страницы приветствия.

Давайте изучим полный код данного файла:

<?php/** * Welcome Page Init * * Welcome page initializer. * * @since 1.0.0 */if ( ! defined( "WPINC" ) ) {    die;}/** * Activates the welcome page. * * Adds transient to manage the welcome page. * * @since 1.0.0 */function wpw_welcome_activate() {    // Transient max age is 60 seconds.    set_transient( "_welcome_redirect_wpw", true, 60 );}register_activation_hook( WPW_PLUGIN_FILE, "wpw_welcome_activate" );/** * Deactivates welcome page * * Deletes the welcome page transient. * * @since 1.0.0 */function wpw_welcome_deactivate() {  delete_transient( "_welcome_redirect_wpw" );}register_deactivation_hook( WPW_PLUGIN_FILE, "wpw_welcome_deactivate" );

Код начинается с документирования данных о файле и проверки ABSPATH. Теперь необходимо добавить и удалить транзитное кэширование при активации и деактивации плагина соответственно.

К счастью, в WordPress имеется два хука для этих целей.

  • register_activation_hook( string $file, callable $function ): этот хук срабатывает при активации плагина. Данная функция принимает имя плагина $file и вызываемую функцию $function в качестве параметров.
  • register_deactivation_hook( string $file, callable $function ): этот хук срабатывает при деактивации плагина. Он принимает принимает имя плагина $file и вызываемую функцию $function в качестве параметров.

Ну а теперь нам надо использовать эти хуки. Я создал временное кэширование для страницы приветствия. Функция set_transient() вызывается из функции wpw_welcome_activate(). В предыдущей статье мы узнали, что оператор set принимает ключ, значение, и срок действия в качестве параметров.

Таким образом, ключ _welcome_redirect_wpw установлен на значение true и время 60 секунд. Этот определяет срок действия кэширования.

Далее хук активации, как мы знаем, этот хук выполняется только тогда, когда плагин активирован. Он добавляет переменную кэширование в базу данных. Как только переменная кэширования добавлена в базу данных, это означает, что мы можем перенаправить пользователя на нашу страницу приветствия.

Далее, я написал код, который выполняется при деактивации плагина. Нам нужна такая функциональность, при которой переменная кэширования будет удалена при деактивации плагина. Для этого я вызываю функцию delete_transient(), которая принимает ключ _welcome_redirect_wpw в качестве параметра. Далее функция wpw_welcome_deactivate(), которую мы добавим в хук деактивации плагина.

Данная часть необязательно, однако, я хочу, чтобы страница приветствия отображалась каждый раз при активации плагина. Если вы этого не хотите, то можете проигнорировать удаление переменной кэширования.

На данный момент мы обсудили часть кода, необходимую для создания и удаления переменной кэширования. Но код страницы приветствия, который перенаправит пользователей на страницу приветствия, пока отсутствует. Мы этим займемся дальше.
