Интро

Мне нравится использовать typography.css в моих проектах; отдельный файл, в котором содержатся все основные типографические стили, которые я использую. Отдельный CSS файл

Разделить ваш CSS файл – отличное решение при создании сайта. Если вы используете препроцессор (Sass, например), тогда разделение проходит наиболее органично. Если же предпочитаете старый добрый читсый CSS, то для вас все еще имеются инструменты, которые помогут вам компилировать ваши файлы в финальный билд.

Примечание: проясним: это не самая лучшая идея иметь множество отдельных CSS файлов в вашем финальном билде (множество HTTP запросов уткнутся в бутылочное горлышко в плане производительности), так что убедитесь перед деплоем, что все ваши файлы объединились в один!

В любом случае, разделение вашего CSS на части означает, что вы сможете использовать какие-то части кода в новых проектах, что делает поддержание очень простым.

Мы создадим typography.css, так что даже если мы используем агрессивный сброс CSS, наша типографика всегда будет на месте.

Кросс-браузерная оптимизация

Шрифты рендерятся различно в зависимости от браузера и платформы. Мы никогда не сможем заставить выглядеть все и везде одинаково, но давайте начнем с оптимизации шрифтов там, где это возможно.

Рендеринг текста

Многие шрифты используют дополнения для увеличения удобства чтения. Лигатуры, кернинг, геометрическая точность обычная не используются браузерами при типичных обстоятельствах, но могут быть использованы, если вы дадите браузеру соответствующие инструкции.

Браузеры на Mac OS постараются обработать шрифты как можно лучше, а различные движки на разных платформах будут отображать только шрифты от 20px и больше (отображать маленькие шрифты точно понятное дело значительно сложней).

Вот CSS:

body  { text-rendering: optimizeLegibility; }

Это говорит браузеру, что, когда возможно, нужно стараться рендерить дополнительные возможности шрифтов, чтобы повысить удобство чтения. Но за это небольшая цена в виде производительности. Если же вам иногда важнее скорость, чем качество, моете дополнительно использовать optimizeSpeed значение на некоторых элементах:

.foo  { text-rendering: optimizeSpeed; }

Где это будет работать? В Mozilla собрали все данные в таблицу, чтобы ответить на это.

Плавность шрифтов

Рендеринг шрифтов по умолчанию в современных браузерах весьма.. слаб. Символы порой отображаются "немного толстыми" (особенно на OS X), из-за чего появляются дополнительные пиксели для смягчения краев. По этой причине многие используют это:

body { -webkit-font-smoothing: antialiased; }

Иначе, вы также можете использовать следующее:

-webkit-font-smoothing: none;-webkit-font-smoothing: subpixel-antialiased;

Как мы сказали, subpixel-antialiased – это, что современные браузеры используют по умолчанию, none выключит смягчение полностью

Если используете antialiased, ваш дизайн станет немного тоньше, но Джеймс Костер уточняет, что формы и кнопки иногда будут игнорировать эти правила.

Для форм и кнопок нужно уточнять:

body,input,button   {-webkit-font-smoothing: antialiased; }

Обновление: как было указано человеком в комментариях, Firefox на OS X теперь имеет собственный префикс, с которым можно поиграть:

p {-moz-osx-font-smoothing: grayscale;}

Это будет иметь схожий с Webkit antialiased эффект. Однако, будьте осторожны: играть со свойствами рендеринга шрифтов может оказаться опасным в долгосрочной перспективе.

Основа

Вернемся к делу, возможно вы начнете ваш типографический шаблон, задав основной размер шрифта, line-height и font-family. Обычно мы используем это на <html> или <body> элемент короткой фразой:

html {font: 100%/1.5 Arial, sans-serif;}

Это установит наш основной размер на 100% от стандартного для браузера размера (обычно 16px), даст нам безъединичное 1.5 для line-height, а затем применит наш шрифт. Вы можете прочитать больше об этом стандартном "100%", прочитав "Еще больше используем em" на Tuts+.

Заголовки

Двигаемся дальше, следующей типографической остановкой станут заголовки. Нужно настроить все шесть, каждый для разного уровня важности.

Чтобы определить, какого размера должны быть ваши заголовки, нам понадобится modular scale. Используем простой набор значений, основанных на normalize.css Николаса Галлахера.

h1 { font-size: 2em;/* 2*16 = 32 */ }h2 { font-size: 1.5em;/* 1.5*16 = 24 */ }h3 { font-size: 1.17em;/* 1.17*16 = 18.72 */ }h4 { font-size: 1em;/* 1*16 = 16 */ }h5 { font-size: 0.83em;/* 0.83*16 = 13.28 */ }h6 { font-size: 0.75em;/* 0.75*16 = 12 */ }

Здесь значения указаны в em, но вы также можете использовать эквивалентные значения в пикселях, упомянутые в комментариях.

Параграфы

Параграфы обычно составляют скелет HTML документа; их стилизация может сделать читабельность лучше, но также может и ухудшить ее. Для начала нужно указать line-heigh и хорошие отступы.

p {line-height: 1.5;margin: 0 0 1em;}

Такой отступ у параграфов даст нам следующее:

Но в печатных публикациях (таких как, ну... книги) чаще встречаются параграфы, начинающиеся с красной строки, чего мы можем добиться, сделав следующее:

.indent p {margin: 0;}.indent p+p {text-indent: 2em;}

Это удалит margin и применит отступ ко всем параграфам кроме первого.

Единственная проблема – нет нижнего margin у последнего параграфа; любой последующий контент намертво приклеится к параграфу. К счастью, это можно решить. Гарри Робертс предлагает интересную технику, которая мне нравится:

p { line-height: 1.5;margin: 0 0 1em;}.indent p+p {text-indent: 2em;margin-top: -1em;}

Здесь вы можете увидеть, как мы применяем стили к параграфам, margin-bottom присутствует на каждом из них. Только в этот раз мы применим одинаковый негативный margin-top на все параграфы, кроме первого.

Разрывы на словах

В эпоху свободных разметок, наши размеры постоянно меняются (ширина строк). В отличие от статичного мира печати, мы не можем вручную набрать разрывы строк, чтобы все было красиво. CSS пока еще плохо поддерживает переносы, но некоторые пытаются размещать их в местах, где они предположительно должны быть. Последствия этого ужасны

К счастью, дело не стоит на месте. Свойство word-break поддерживается Chrome и старыми версиями IE, плюс дальнейшая поддержка не за горами вместе с переносами для CSS3. Особенности этого всего хорошо рассмотрены Кеннетом Ошенбергом, но на текущий момент, вот вам кросс-браузерные переносы CSS3 во всей красе:

p {word-break: break-all;/* Non standard for webkit */word-break: break-word;-webkit-hyphens: auto;-moz-hyphens: auto;-ms-hyphens: auto;hyphens: auto;}

Обновление: вновь, спасибо, что указали на это; IE8 поддерживает word-break без всяких префиксов. Зайдите на developer.mozilla.org, чтобы узнать больше.

К якорям!

Как вы думаете, как выглядят гиперссылки? По умолчанию большинство браузеров будут подчеркивать активные ссылки. Это настолько стало нормой, что вам лучше придумать очень хорошую причину, чтобы сделать иначе. Пользователи ожидают, что подчеркнутые слова будут ссылками, так что не подчеркивайте случайны слова ради акцентирования.

Также стоит упомянуть о статусе :visited (другой цвет сработает неплохо, но тут много чего и не сделаешь) и для ссылок :hover

a,a:visited {text-decoration: underline;}a:hover{ text-decoration: none; }

Некоторые утверждают, что подчеркивание ухудшает читаемость ссылки, поэтому вместо подчеркивания можно использовать border-bottom, который увеличит пространство между строкой и буквой.

Также, (как подсказал мне Bowie в комментариях) вы можете включить псевдо-классы :active и :focus. Они не особо полезны для мыши (почему я и забываю о них иногда), но для клавиатуры либо других технологий помощников они в самый раз.

Примечание: вы можете даже включить :focus:hover, если чувствуйте себя несколько одержимым!

/* states for accessibility http://meyerweb.com/eric/css/tests/css2/sec05-11-03.htm */a:active{ color: tomato; }a:focus{ border: 1px dotted tomato; }

Последнее замечание. Раз мы обсуждаем якоря, хорошо здесь же упомянуть о стилизации различных ссылок. Если хотите, вы можете использовать селекторы CSS для стилизации ссылок различными способами, например, ссылки на скачивание:

a[href$=".zip"] { color: tomato; }

Предположения

Если вы обнулили все ваши стили, то бразуер вас подстрахует. Но всегда лучше иметь свою основу. Выдерживают ли ваши strong? А em акцентированы?

strong {font-weight: bold;}em {font-style: italic;}

Шаблоны кода

Мы очень часто используем элемент <code> на Tuts+, так что я точно всегда про него помню, но используете ли вы эту настройку?

code,pre {font-family: monospace, serif;    font-size: 1em;}

Отлично. Наш хороший друг шрифт Monospace имеет в себе легко различимые символы, поэтому он и используется в редакторах кода по всему миру. Мы включаем селектор <pre> тоже; "code" используется для строчных шаблонов, "preformatted" используется для блоков кода.

Цитаты

Вам не нужно усердствовать с вашими <blockquote> элементами, но отделить их от остального вашего контента важно. Они создают ритм статье и добавлят интереса.

Итак, чтобы не погружаться глубоко в стили здесь, давайте всего лишь выделим наши blockquotes:

blockquote { font-style: italic; }blockquote:before {  content: "\201C"; display: inline-block; padding-right: .4em;}

Это сделает нашу цитату выделенной курсивом (будем надеяться, что выбранный шрифт иметь вариант с курсивом, иначе за дело возьмется браузер), а затем используем псевдо элемент для создания символа цитаты.

Просто. Этот символ цитаты \201C будет всегда динамично вставляться в наши цитаты. Дальнейшая стилизация полностью на ваше усмотрение!

Стили для печати

Стилизация под экраны это одно, но как часто вы задумывались о том, что происходит, когда пользователь нажимает "Напечатать"? ы стараемся вообще не думать о чем-либо печатном.

Взяв немного из HTML5, мы можем применить несколько стилей для тех, кому важно читать с бумаги. Начнем с элемента media, в который будем все вкладывать:

@media print {}

Раскрасьте в черный

Теперь давайте очистим все, сделав так, чтобы оно печаталось полностью черным, удалив все задние фоны (мы не хотим тратить краску на это) и дополнительные тени. И да, мы будем использовать !important, который довольно полезен в этом случае и который ничего не будет перекрывать.

@media print {    * {background: transparent !important;        color: #000 !important; /* Black prints faster: h5bp.com/s */        box-shadow: none !important;        text-shadow: none !important;}}

Страницы

Еще кое-что мы возьмем из HTML5BP, что повысит удобство чтения с бумаги. Опять же, вложим это в наши @media print, взгляните:

 p,    h2,    h3 {orphans: 3; /* http://css-tricks.com/almanac/properties/o/orphans/ */        widows: 3; /* http://css-tricks.com/almanac/properties/w/widows/ */}

Эти свойства применяются только к контенту, который поделен на страницы и никак не повлияют на содержимое на экране. Они контролируют указание страниц у документов, предотвращая появление линий наверху новых страниц, либо в самом низу старых.

Крис Койэр неплохо об этом рассказывает.

widows = минимальное количество строк в абзаце на новую страницу.
orphans = минимальное число строк в абзаце на старую страницу.

Крис Койэр

Например, widows – это последние строк абзаца, которые иногда залезают на новую страницу.

В то время как orphans – это начальные строки абзаца в конце страницы:

Дробление абзаца (или заголовка) нарушает поток чтения, так что правила CSS, упомянутые выше делают так, чтобы у нас было как минимум по три строки в любом из случаев. Если так не получается, то widow переносятся на следующую страницу, как и orphans, в общем-то.

Разрывы страниц

И напоследок, давайте предупредим разрыв абзаца и заголовка (который выглядит так):

Вновь, нам понадобится следующее в наших @media print.

    h2,    h3 {page-break-after: avoid;}

page-break-after может принимать множество значений. Мы использовали avoid, который не позволяет странице закончиться на заголовке (перемещая заголовок на новую страницу). Будьте осторожны, это свойство однажды пересечется с break-after.
